🔢 الـ Math Implementation Details
══════════════════════════════════════════════════════════════════════

A. حساب الـ Running Balance (CRITICAL — لازم تتفادى float drift)

   ⚠️ استخدم r2() من utils/format.js على كل عملية حسابية. CLARK كان
   عنده bug تاريخي قبل V19.66 بسبب float accumulation (راجع invoices.js
   header comment).

   ```js
   function buildStatement(allEntries, openingBalance, sideConvention) {
     // sideConvention = "sales" | "purchase"
     // allEntries already sorted by date ASC then by createdAt ASC
     let balance = r2(openingBalance);
     const rows = [];

     for (const entry of allEntries) {
       const { debit, credit } = computeDebitCredit(entry, sideConvention);
       balance = r2(balance + debit - credit);
       rows.push({
         ...entry,
         debit: r2(debit),
         credit: r2(credit),
         balance,
       });
     }

     return rows;
   }
   ```

B. الـ Entry types & their debit/credit mapping

   نوع الحركة (Sales side):
     - sales_invoice_subtotal:    debit = invoice.subtotal, credit = 0
     - sales_invoice_discount:    debit = 0, credit = invoice.discount
     - sales_credit_note:         debit = 0, credit = creditNote.total
     - customer_payment:          debit = 0, credit = payment.amount

   نوع الحركة (Purchase side — معكوسة):
     - purchase_invoice_subtotal: debit = 0, credit = invoice.subtotal
     - purchase_invoice_discount: debit = invoice.discount, credit = 0
     - purchase_debit_note:       debit = debitNote.total, credit = 0
     - supplier_payment:          debit = payment.amount, credit = 0

C. الترتيب الزمني (Sort order)
   1. Primary: entry.date (YYYY-MM-DD)
   2. Secondary: entry.createdAt (ISO timestamp) — للـ multiple entries
      في نفس اليوم
   3. Tertiary: لو فاتورة بـ سطرين (subtotal + discount):
      always subtotal first، discount second (sub-order ثابت)

D. حساب الـ Opening Balance (لو from-date محدد)

   ```js
   function computeOpeningBalance(allEntries, fromDate, sideConvention) {
     const before = allEntries.filter(e => (e.date||"") < fromDate);
     let balance = 0;
     for (const e of before) {
       const { debit, credit } = computeDebitCredit(e, sideConvention);
       balance = r2(balance + debit - credit);
     }
     return balance;
   }
   ```

   ⚠️ ده لازم يكون deterministic — لو الـ user filtered بـ "نوع الحركة"
   (فواتير فقط مثلاً)، الـ opening balance يفضل يحسب من **كل** الحركات
   (مش بس الـ filtered) لأنه ده الرصيد الحقيقي قبل الفترة. الـ type
   filter يطبق على rows في الـ range بس.

E. إدارة الـ status

   ⚠️ الفواتير الـ draft (status=draft) و void (status=void) — هل تظهر في الكشف؟
       Recommended: 
         - draft: لا تظهر (لسه ما اتـ post-تش محاسبياً)
         - void: لا تظهر (ملغية — مش حركة فعلية)
         - posted فقط هي اللي تظهر
       اطلب من Ahmed التأكيد قبل ما تـ implement.

   ⚠️ الـ payments الـ pending (لو CLARK عنده pending state — راجع schema):
       Recommended: تظهر بس بـ visual indicator (italic أو grayed)
