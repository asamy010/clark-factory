C. Stock Effective Calculation

  function getEffectiveStock(order, data) {
    const confirmed = getConfirmedStock(order);
    const broken = getBrokenStock(order);
    
    /* Subtract any active reservations */
    const activeReservations = (data.customerReservations || [])
      .filter(r => 
        ["pending", "approved"].includes(r.status) &&
        r.lockExpiresAt > nowISO()
      );
    
    let reservedSeries = 0;
    let reservedBroken = 0;
    
    activeReservations.forEach(r => {
      r.items.forEach(item => {
        if(item.orderId === order.id){
          reservedSeries += item.requestedSeries || 0;
          reservedBroken += item.requestedBroken || 0;
        }
      });
    });
    
    return {
      series: Math.max(0, confirmed.series - reservedSeries),
      broken: Math.max(0, broken - reservedBroken),
      total: Math.max(0, (confirmed.series + broken) - reservedSeries - reservedBroken),
    };
  }

  ⚠️ Hide model from catalog لو effective.total === 0

J. Race Condition Prevention

  Firestore transactions:
  
  async function reserveStock(items, transaction) {
    // Read current state
    const config = await transaction.get(factoryConfigRef);
    
    // Validate stock for each item
    for(const item of items) {
      const order = config.orders.find(o => o.id === item.orderId);
      const effective = getEffectiveStock(order, config);
      
      if(effective.series < item.requestedSeries ||
         effective.broken < item.requestedBroken) {
        throw new Error(`stock_insufficient:${item.modelNo}`);
      }
    }
    
    // Atomic write: add reservation
    transaction.update(factoryConfigRef, {
      customerReservations: [...config.customerReservations, newReservation]
    });
  }

## 💡 ملاحظات لـ Ahmed قبل التنفيذ

**ده فيتشر كبير (3-4 أسابيع للـ MVP الكامل)** بـ ROI هائل:
- يفك ضغط الأدمن (العملاء يطلبوا بدون مكالمات)
- يفتح قناة customer acquisition جديدة (الكتالوج العام)
- يجمع بيانات العملاء الجدد بشكل احترافي
- يدي business intelligence (إيه الموديلات اللي بـ يتم browsing-ها مش بـ يتم booking-ها)

**3 أسئلة فقط متبقية للـ AI يسألك (معظم القرارات اتأخدت):**

1. **WhatsApp Bridge stability:** الـ Bridge متاح ومستقر؟ — critical dependency للـ OTP
2. **URL routing:** `/?portal=1&catalog=1` أم `/catalog` (clean URL)؟ — توصيتي الثانية لو الـ Vite routing بـ يدعم
3. **Catalog PDF:** server-side (puppeteer) أم client-side (jsPDF)؟ — server أحسن لكن أعقد

**أكتر 3 مخاطر تقنية لازم تعرفها:**

1. **Race Condition على الـ Stock:**
   - عميلين يحجزوا نفس الموديل في نفس الثانية
   - الـ Firestore transactions بـ تـ handle ده، لكن لازم implementation دقيق
   - الـ AI لازم يستخدم `runTransaction` في كل reserve operation

2. **WhatsApp Bridge Ban Risk:**
   - لو OTP بـ يـ spam، الـ Bridge ممكن يـ ban من WhatsApp
   - Rate limiting strict:
     * Max 3 OTP/phone/hour
     * Max 10 OTP/IP/hour
     * Cooldown بين الـ resends (30 ثانية)

3. **Privacy + GDPR-style:**
   - بتجمع بيانات عملاء جدد (name, phone, address)
   - لازم يكون فيه "checkbox" قبل الـ submission: "أوافق على معالجة بياناتي"
   - Privacy policy page بسيطة

**نقطة معمارية حرجة:**

الـ flow بعد الـ approval:
```
Admin clicks "Approve" 
  → If new customer: create entry في data.customers
  → Status: "approved"
  → Stock يفضل locked

Admin clicks "Convert to Sale" (separate action)
  → Create customerDelivery session بـ pending status
  → Status: "converted"
  → Stock locks تـ released (الـ delivery يـ track الـ stock من دلوقتي)
```

ده **two-step** بدلاً من approve = auto-convert. السبب:
- الـ approve يقول "موافق على الـ deal"
- الـ convert يقول "ابدأ التنفيذ"
- ممكن يكون في فجوة بينهم (مفاوضات، تنسيق، إلخ)

لو حابب one-step (approve = convert فوراً)، قول للـ AI في الـ Phase 1.
