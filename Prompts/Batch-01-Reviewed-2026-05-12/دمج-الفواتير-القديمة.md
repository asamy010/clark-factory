C.2 — حالات الـ session (حرج للفهم):
  
  | حالة الـ session | الـ behavior في الـ tool |
  |-----------------|--------------------------|
  | كل الفواتير draft | ✅ قابل للدمج بالكامل (default selected) |
  | بعضها posted، بعضها draft | ⚠️ "mixed" — لا تـ select، خطر على المحاسبة |
  | كلها posted | 🔒 read-only — لا تـ touch (لها قيود محاسبية) |
  | فيها void | تجاهل الـ void، اعتبرها كأنها مش موجودة |

C.3 — Merge Logic (server-side أفضل — admin SDK)
  
  أنشئ endpoint جديد: `/api/admin/merge-fragmented-invoices.js`
  
  Body: { 
    sessionIds: ["sess_abc", "sess_xyz", ...],
    type: "sales" | "purchase",
    dryRun: bool,
  }
  
  Logic:
    1. verifyAdminToken
    2. لكل sessionId:
       a. اقرأ كل invoices اللي deliveryRefs[].sessionId === sessionId
          (للـ sales) أو receiptRefs[].sessionId (للـ purchase)
       b. لو أي واحدة status="posted" → skip (لا تـ touch)
       c. لو كلها draft:
          - الـ "keeper" = الأقدم (أصغر invoiceNo) — يـ keep الـ number
          - الباقي = "victims" — يـ merge في الـ keeper
          - For each victim:
            * Concatenate الـ items[] (مع dedup لو نفس orderId+price)
            * Append deliveryRefs[]
            * Recompute subtotal + discount + total بـ r2()
          - Delete الـ victim invoices من الـ array
          - Update keeper's items + refs + totals
       d. سجّل في migrationLog/{id}:
          {
            type: "invoice-merge-v21.x.y",
            sessionId,
            keptInvoiceNo,
            mergedInvoiceNos: [...],
            mergedAt, mergedBy: auth.email,
            backup_doc_id: "pre-merge-..."
          }
    3. ⚠️ قبل أي merge: backup كامل للـ factory/config لـ
       backups/pre-invoice-merge-v21.x.y-{ts}
    4. Idempotent: لو الـ session اتدمجت قبل كده (مفيش fragmentation)،
       تخطي بدون error

C.4 — Counter handling (حرج)
  لما الـ "victims" تتشال:
    - أرقام الفواتير المحذوفة بتـ "تضيع" (gap في الـ sequence)
    - ده مقبول محاسبياً لـ drafts (لسه ما اترحلتش)
    - لكن لو كان counter at 100 وحذفنا 4، التالي بيكون 101 (مش 97)
    - **مفيش re-numbering** — الـ counter لا يـ rollback (idempotent + safe)
    - الـ admin يـ note في الـ migration log إن دي الـ numbers اللي اتشالت

C.5 — Safety Checks
  ❌ Refuse to merge لو:
    - أي invoice في الـ session status="posted"
    - الـ customerId مش متطابق بين كل invoices الـ session (سيناريو غريب)
    - الـ session غير موجودة في الـ data
    - مفيش deliveryRefs[] على الـ invoices (legacy pre-V18.65)
  
  ✅ Allow merge لو:
    - كل invoices draft + same customerId + valid sessionId

C.6 — Confirmation Flow
  Dry-run mode بيـ run أولاً، يـ show:
    "هـ يتم دمج 8 توزيعات (23 فاتورة → 8 فاتورة). 
     هـ نوفّر 15 رقم في الـ sequence.
     Backup هـ يتعمل أوتوماتيك.
     تأكيد؟"
  
  لو yes → الـ actual merge
