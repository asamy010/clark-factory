## 🔍 الفهم الصحيح للمشكلة من الكود

### المشكلة الأولى: تعديل قيود اليومية

**اللي موجود حالياً (positive):**
- `JournalEntryModal.jsx` بيقبل `existing` prop — يعني عنده infrastructure للـ edit
- `postEntry()` في `posting.js` السطور 108-119 بيـ update existing entry لو الـ `(sourceType, sourceId, date)` متطابق
- `reverseEntry()` السطور 142-185 بيعمل void + reversal entry (correction pattern)

**الـ Gap:**
- كل القيود بتدخل بـ `status: "posted"` مباشرة (السطر 130 من posting.js)
- مفيش `status: "draft"` للقيود
- معنى ده: لو الـ admin عاوز يصحح قيد، حالياً عنده خيارين:
  1. `reverseEntry` يخلق قيد عكسي + يـ mark الأصلي void (audit trail ضخم)
  2. Manual edit عبر `postEntry` بـ نفس sourceId (لكن مفيش UI واضح ده)
- المطلوب: تحويل لمسودة → تعديل → re-post (workflow أنظف للأخطاء البسيطة)

### المشكلة الثانية: الربط الفرعي بالطرف (Subsidiary linkage)

**اللي موجود حالياً (positive):**
- `partyId` + `partyName` fields في line schema (`posting.js` السطر 74-75) — optional
- `PartyLedgerTab.jsx` (217 سطر) موجود — كشف حساب الطرف من قيود اليومية
- `getPartyLedger()` في `aggregate.js` السطر 192 بيـ filter بـ `line.partyId === partyId`

**الـ Gap الحرج:**
- `partyId` **optional** ومش enforced — في `validateLines()` (السطر 30-54) لا فيه أي check
- مفيش flag في CoA: "ده control account يستلزم طرف فرعي"
- `JournalEntryModal` مش بيـ render PartyPicker لما الحساب يكون subsidiary
- النتيجة العملية: لو المحاسب عمل قيد يدوي من "1210 عملاء" لـ "1130 شيكات تحت التحصيل":
  - السطرين بيدخلوا بدون partyId
  - `getPartyLedger()` بيـ filter بـ partyId → السطر مش بيظهر في كشف العميل
  - **الشيك ضايع من حساب العميل الفرعي** ← ده اللي أحمد بيشتكي منه
- `PartyLedgerTab` يدعم بالفعل: customer / workshop / employee — لكن **مفيش supplier!** ده gap إضافي

**الحسابات اللي لازم تكون "بـ ربط فرعي إجباري":**

| الحساب | partyType الإجباري |
|---|---|
| 1210 عملاء | customer |
| الموردين (1110 / 2110 حسب الـ CoA) | supplier |
| 1130 شيكات تحت التحصيل | any (customer أو workshop) |
| 2200 شيكات الدفع | any (supplier أو workshop أو employee) |
| 2300 ورش (دائنين ورش) | workshop |
| 1240 سلف موظفين | employee |
| 2400 رواتب مستحقة | employee |
| 2310 مديني ورش (مقدمات) | workshop |

📋 PART A — Journal Entry Editing

A.1 — Status workflow الجديد

  الحالات الحالية (في الكود):
    "posted" → entry معتمد
    "void"   → entry ملغى

  الحالات الجديدة:
    "draft"  → entry قابل للتعديل (مش بيدخل في الـ aggregations)
    "posted" → معتمد ومـ included في كل الحسابات
    "void"   → ملغى (legacy)

    posted → draft  (admin: "✏️ تحويل لمسودة")
    draft → posted  (admin: "💾 حفظ وترحيل")
    draft → DELETE  (admin: "🗑 حذف المسودة")
    posted → void   (via reverseEntry — موجود)

  ⚠️ التعديل المباشر للـ posted entry ممنوع. لازم draft الأول.
  ⚠️ Auto-posted entries (sourceType !== "manual") لا تـ allow تحويل

## 💡 ملاحظات لـ Ahmed قبل ما تستخدم الـ برومبت

**الفيتشر ده 2-3 أسابيع شغل ومستقل عن الباقي.** قيمته الكبيرة محاسبياً: بعده مفيش "شيكات ضايعة" من حسابات العملاء، وبتقدر تطابق إجمالي 1210 (عملاء) مع مجموع الكشوف الفرعية.

**3 نقاط الـ AI هيسأل عنها — جهز إجاباتك:**

1. **Suppliers data:** عندك `data.suppliers` في configDoc ولا في مكان مختلف؟ افحص قبل أو خلي الـ AI يستكشف.

2. **CoA codes:** لو الـ codes الفعلية في CLARK بتاعك مختلفة عن الـ defaults (مثلاً عميل = 12 مش 1210)، الـ migration هيـ match by **الاسم** مش الـ code. أكد الأسماء العربية اللي بتستخدمها.

3. **"any" partyKind للشيكات:** لما تعمل قيد على "شيكات تحت التحصيل" والـ partyKind="any"، تفضّل:
   - (أ) Picker للنوع أولاً (customer/workshop/إلخ) ثم الـ party
   - (ب) Dropdown واحد فيه كل الـ parties مع icon للنوع
   
   توصيتي: (أ).

**نقطة حرجة:** البرومبت بيمنع تحويل الـ auto-posted entries لـ draft عبر الـ UI. ده اختيار محسوب — لأن لو حولت قيد فاتورة مبيعات لـ draft وعدلت فيه، الفاتورة الأصلية تبقى out-of-sync. الحل الصحيح: تعدل الفاتورة، الـ autoPost يـ re-run. لو عاوز override (admin طوارئ)، قول للـ AI يضيف override mode مع warning قوي.

**ترتيب التنفيذ:**
الـ Slices 1-2 بيحلوا "تعديل القيود" لوحدهم. الـ 3-8 بيحلوا "الربط الفرعي". تقدر تنفذ 1-2 الأول وتاخد breathing room.
