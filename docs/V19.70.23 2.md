# V19.70.23 — Approach A applied to bulk delivery PDF

**Date:** 2026-05-06
**Type:** fix (architectural)
**Status:** shipped — pending user verification

## The breakthrough insight

The user observed:
- **Browser print preview** renders Arabic correctly ✓
- **Browser "Save as PDF"** from the print dialog produces a correct Arabic PDF ✓
- **html2canvas-generated PDF** breaks Arabic shaping ✗

This proves the bug is NOT in the HTML or the fonts — it's specifically in html2canvas's offscreen canvas rendering pipeline. Chrome's native print → PDF works perfectly.

## What changed in V19.70.23

The bulk delivery WhatsApp PDF (the main user-facing case) is migrated from the html2canvas pipeline to the jsPDF + embedded Cairo TTF + Arabic shaper pipeline (Approach A from the V19.70.14 handoff).

### Before

```
buildOneCustomerHTML(c) → HTML string
htmlToPdfBase64(html)   → html2canvas captures into <canvas>
                       → toDataURL("image/jpeg")
                       → embed JPEG in PDF
                       → return base64
```

Result: image-based PDF. Image-based = pixelated when zoomed, larger file, broken Arabic shaping.

### After

```
buildOneCustomerPayload(c) → structured object
buildDeliveryReceiptPdfBase64(payload)
   → new jsPDF instance
   → addFileToVFS('Cairo-Regular.ttf', base64)
   → addFont(...)
   → setR2L(true)
   → pdf.text(arNoEmoji(text), x, y, {align})
   → pdf.autoTable({ head, body, foot, styles: { font: 'Cairo' } })
   → pdf.addImage(qrDataUrl, 'PNG', ...)
   → return base64
```

Result: vector PDF. Vector = crisp at any zoom, smaller file, Arabic shaped via embedded shaper using Cairo's presentation forms.

## Receipt layout in the new PDF

The layout closely mirrors the HTML print version:

1. **Top header row** — factory logo (right), factory name + sub-line (right of logo), receipt title box "اذن تسليم — {customer}" + date/time (left, in a sky-50 rounded box)
2. **Header underline** — sky-600 line
3. **Section heading** — "اذن تسليم عميل" (truck emoji stripped) — sky-600 bold + thin underline
4. **Customer info table** — autoTable, two rows × six columns (label/value × 3 pairs): العميل/name, التليفون/phone, التاريخ/date, العنوان/address. Label cells have a slate-200 background.
5. **Section heading** — "تفاصيل الاستلام"
6. **Items table** — autoTable with head, body (one row per item), and foot (aggregation row). Columns: الموديل · الوصف · الكمية · السعر · الإجمالي. When `noPrices` is set, السعر + الإجمالي columns are hidden.
7. **Discount block** (when `discPct > 0` and prices shown) — black-bordered box with three lines: الإجمالي قبل الخصم (slate-900), خصم N% (red-500), الصافي المستحق (emerald-600 bold/larger).
8. **QR block** (when `qrDataUrl` provided) — sky-500 dashed border, QR image on the left (26×26mm), title + description on the right.
9. **Signature row** — two lines, two labels: توقيع العميل (right) + customer name underneath, مسؤول التسليم (left) + receiver name underneath if provided.
10. **Footer** — slate divider + factory name on the right + date · Powered by CLARK on the left, on the LAST page only.

Page breaks happen automatically inside autoTable for long item lists. The discount block, QR, and signatures are positioned via `pdf.lastAutoTable.finalY + N` and trigger an explicit `pdf.addPage()` if remaining vertical space is insufficient (~70mm needed for QR + signatures + footer).

## Arabic shaping (recap)

The shaper in `arabicPdf.js` (~150 lines, V19.70.21) implements the standard Arabic contextual joining algorithm:
- Per-letter joining type table (right-joining vs dual-joining) for the full Arabic Unicode range U+0621–U+064A
- Lam-Alef ligature collapse (ل + ا → single glyph)
- Output reversed at the end so visual order matches RTL layout

Cairo TTF includes the full Arabic Presentation Forms-B block (U+FE70–U+FEFC) so the shaper's output codepoints have valid glyphs. The font is embedded in the PDF binary via `pdf.addFileToVFS` so there's no external font dependency at view time.

## Failure modes + fixes

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| Some letters appear as boxes (□) | Cairo's cmap doesn't have a presentation form codepoint we generated | Add the missing letter to `_AR_FORMS` in `arabicPdf.js`, OR switch to a font with full PFB coverage like Amiri |
| Letters disconnected | The shaper picked the wrong joining form | Check `_AR_FORMS[codepoint]` for that letter — verify the joining-type flag |
| Whole text mirrored | `out.reverse()` + `setR2L(true)` are double-applied for that string | Remove the `setR2L` flag for the offending text section |
| Cairo TTF download fails | jsdelivr/gh CDN is down | Switch the URL in arabicPdf.js to an alternate mirror, or bundle the TTF as a static asset in `public/fonts/` |
| Arabic-Indic numbers don't render | jsPDF text layout limitation | Use Latin numerals (en-US locale, current default in `_fmtNum`) |

## Path forward

- **If the PDF renders correctly** → the bug is finally fixed. The `includePdf` toggle stays default ON. Future PDFs (other features) can use this same engine.
- **If specific glyphs render wrong** → small targeted fixes to `_AR_FORMS` table.
- **If the whole approach is still broken (unlikely given the architecture is fundamentally sound)** → fall back to Approach B (server-side Puppeteer Vercel function).
