## Issue 1: Portal 403 لرابط عميل معيّن

### السياق

User report:
> "رابط غير صالح لعميل معين. باقي العملاء كلهم شغّالين تمام، بس عميل محدد بيدّيني 403."

Console showed:
```
GET /api/customer-portal?c=moiheq7u-4nrminfxc9bi&sig=1CVylEjtJ-_FC13E
HTTP 403
```

But the URL the customer clicked had `t=1778151917` parameter — it just never reached the API call.

### السبب

V19.64 (months ago) added timestamped HMAC signatures (`signCustomerIdWithTs`)
for portal links — `?p=c&i=...&t=<unix>&s=<sig>`. The API endpoint at
`api/customer-portal.js` correctly handles both:
- New format with `t` → V2 timestamped verification
- Old format without `t` → legacy fallback

But the **frontend** (`src/App.jsx` + `CustomerPortalPage.jsx` +
`WorkshopPortalPage.jsx`) was never updated to forward `t` to the API:

```js
// App.jsx (BEFORE):
const c = urlParams.get("c") || urlParams.get("i");
const sig = urlParams.get("sig") || urlParams.get("s");
// ← missed: const t = urlParams.get("t");
return <CustomerPortalPage params={{c, sig}}/>;

// CustomerPortalPage.jsx (BEFORE):
const url = "/api/customer-portal?c=" + c + "&sig=" + sig;
// ← no &t=...
```

So the API received only `c` + `sig`. When `t` is absent, the API uses legacy
verification — works for OLD links (signed with the legacy scheme), fails for
NEW links (signed with the V2 timestamped scheme):

```
verifyCustomerSig(custId, sig, ts=undefined)
  → falls into legacy path: signCustomerId(custId)
  → "portal:" + custId  (no ts in HMAC payload)
  → mismatch with the V2 sig that was generated as "portal:v2:" + custId + ":" + ts
  → 403
```

This worked for older customers because their links were generated when the
legacy scheme was active. The customer who failed had a link minted post-V19.64.

### الإصلاح

**`src/App.jsx`** — read `t` and forward to portal page:
```js
const c   = urlParams.get("c")   || urlParams.get("i");
const sig = urlParams.get("sig") || urlParams.get("s");
const t   = urlParams.get("t")   || "";  // ← V19.78.2
return <CustomerPortalPage params={{c, sig, t}}/>;
```

**`CustomerPortalPage.jsx`** + **`WorkshopPortalPage.jsx`** — destructure `t`
and append to API URL:
```js
const { c: custId, sig, t: ts } = params;

const url = "/api/customer-portal?c=" + encodeURIComponent(custId) +
            "&sig=" + encodeURIComponent(sig) +
            (ts ? "&t=" + encodeURIComponent(ts) : "");
```

The `(ts ? "&t=..." : "")` guard means old links without `t` continue to work
unchanged (they hit the API's legacy path as before).

## Issue 2: Agent يلتزم بالعامية المصرية

### السياق

User report:
> "عاوزه يتكلم لهجة مصرية عامية، عشان بيتكلم عربي خليجي أحياناً.
> وعاوز التحكم الكامل في الشخصية من الـ UI."

The system prompt had a single weak instruction:
> "- مهني وودود — عامية مصرية مهذبة"

Claude can drift to Khaleeji ("شلونك", "تبي", "الحين") under that weak
guidance, especially in longer conversations.

### الإصلاح

**1. Hard rule #0 in the golden rules section** — explicit forbidden-word
table covering the most common Khaleeji slips:

```
**0. اللهجة المصرية العامية حصراً — هذه قاعدة صارمة لا تكسر أبداً.**

| ❌ ممنوع              | ✅ المصري               |
| كيف حالك / شلونك     | إزيك / إزي حضرتك       |
| ماذا / ماذا تريد      | إيه / عاوز إيه          |
| الآن / الحين          | دلوقتي                   |
| تبي / هل تريد         | عاوز / تحب              |
| نعم / أجل / إي        | أيوه / تمام             |
| لا / كلا              | لأ                       |
| أين / وين             | فين                      |
| متى                  | إمتى                     |
| لماذا / ليش           | ليه                      |
| كم / كم سعر           | بكام                     |
| هذا / هذي             | ده / دي                  |
| رائع / ممتاز           | تمام / حلو              |
| حسناً                  | تمام / ماشي / حاضر       |
... (full table in src/agent/prompts/system_base.md)
```

**2. Personality.language wired into the prompt** — the UI selector
(egyptian_polite / msa / bilingual) was previously decorative; now it
substitutes into the prompt:

```js
// llm.js → buildPersonalityExtras():
const langMap = {
  egyptian_polite: "🇪🇬 **اللهجة المصرية العامية حصراً** — راجع الجدول أعلاه. ممنوع الخليجي/الفصحى/الشامي.",
  msa:             "🌐 العربية الفصحى رسمياً — لا عامية.",
  bilingual:       "العميل قد يستخدم العربية أو الإنجليزية — رد بنفس اللغة.",
};
```

**3. Personality.emojiUse wired** — same treatment (none/minimal/moderate/rich
→ explicit emoji limit).

**4. Admin's systemPrompt framed as authoritative override:**
```
═══ تعليمات إضافية من الإدارة (إلزامية — لها الأولوية) ═══
{personality.systemPrompt}
═════════════════════════════════════════════════════════
```

So if the admin types "اكتب رسائل قصيرة جداً، 1 جملة بس" or "استخدم اللهجة
الصعيدية" — those instructions take priority over the generic style hints.

### Full UI control

The Personality tab in CLARK gives the admin complete control:
- Identity: name, language, style, answer length, emoji use
- Phrases: custom greetings + closings (multi-tag)
- Forbidden: any phrases the admin wants the agent never to use
- **System Prompt**: free-form instructions appended verbatim with
  highest priority

Every change saves to `factory/config.aiAgent.personality` and is read on
every LLM call (60s TTL cache), so admin edits propagate within ~1 minute.
