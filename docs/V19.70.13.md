# V19.70.13 — Bulk WhatsApp delivery: match per-row format exactly

**Date:** 2026-05-06
**Type:** fix, ux
**Status:** shipped

## Why this version exists

V19.70.12 shipped a working bulk-send, but the user reported two regressions versus the existing per-row print/WA flow:

1. **The PDF used a simplified layout** different from the existing 🖨 button output: "البي دي اف عاوزه يكون نفس نسخة الاستلام الموجوده بالفعل."
2. **The WhatsApp text didn't match** the existing per-row 📱 button format: "كمان رسالة الواتس تكون نفس الرسالة بالظبط اللي بتوصل العميل."

V19.70.13 brings both into exact parity.

## PDF: full structure parity

`buildOneCustomerHTML` was rewritten to mirror the per-row receipt structure:

1. **CLARK header** — logo + factory name + sub-line + title `'اذن تسليم — {customer}'` + date/time.
2. **العميل/التليفون/التاريخ/العنوان** info table.
3. **`'تفاصيل الاستلام'` table** with الموديل / الوصف / الكمية / السعر / الإجمالي + aggregation row.
4. **Discount block** — الإجمالي قبل الخصم / خصم N% / الصافي المستحق.
5. **QR code section** with `'تأكيد الاستلام'` note.
6. **Signature row** — مسؤول التسليم / توقيع العميل.
7. **Factory branding footer**.

## WhatsApp message: exact format match

The new builder uses the same format as the per-row 📱 button:

```
*CLARK — اذن تسليم عميل*

• العميل: {name}
• التاريخ: {date}
─────────────────
• *{modelNo}*: {qty} قطعة × {price} = {lineTotal} ج.م
• *{modelNo2}*: ... 
─────────────────
الاجمالي {totalQty} قطعة
الاجمالي {totalMoney} ج.م
خصم {discPct}%: -{discAmt} ج.م
الصافي المستحق: {netAmt} ج.م

📱 *برجاء مسح كود QR في إذن التسليم للتأكيد باستلام البضاعة كاملة*

{optional account summary footer if printSettings.whatsappSummary is on}
```

The optional account-summary footer comes from `formatCustomerSummaryWA(buildCustomerSummary(...))`.

## Reliability fix: QR embedded as data URL

Before V19.70.13 the HTML contained `<canvas data-qr=...>` + a CDN script for `qrcode.js` that ran after page load. In an offscreen html2canvas capture, the canvas could be empty because the CDN script wasn't loaded yet.

**Fix:** use the `qrcode` package (already in dependencies) to pre-generate the QR as a data URL synchronously:
```js
const qrDataUrl = await QRCode.toDataURL(url, { width: 200, errorCorrectionLevel: 'M' });
```
Then embed as `<img src='data:...'/>`. No race condition, the PDF always has the QR.

Side effect: `buildOneCustomerHTML` is now async. The send loop awaits correctly. No performance impact since the loop is already sequential.
