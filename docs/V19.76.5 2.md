## السياق

User request:

> "عاوز اضيف في تريجر الفورية دفعة كاش لمورد بنفس طريقة دفعة كاش لعميل"

الـ Triggers الفورية كان عندهم `paymentReceived` للعميل (دفعة كاش/تحويل من
عميل) لكن مكنش فيه mirror للمورد. لما المصنع يدفع كاش/تحويل لمورد، الـ event
ما كانش يفير. الشيكات للمورد ليها `checkPaymentIssued` بالفعل.

## الإصلاح: Event جديد `supplierPaymentSent`

نفس بنية `paymentReceived` بالظبط، بس supplier-side. مضاف في 6 أماكن للـ
end-to-end coverage:

### 1. `src/utils/automation/eventBuilder.js` (و الـ mirror في `api/_eventBuilder.js`)

- `EVENT_VARIABLES.supplierPaymentSent`:
  ```js
  {
    label: "💸 دفعة كاش/تحويل لمورد",
    description: "نقدي/محفظة/انستاباي/تحويل بنكي (تسجيل supplierPayment) — مش شيكات",
    detection: "client (instant) + cron fallback",
    recipientRoles: ["supplier", "owner"],
    variables: {
      supplier: ["{supplierName}", "{amount}", "{method}", "{balance}", "{date}"],
      owner:    ["{supplierName}", "{amount}", "{method}", "{balance}", "{date}", "{office}"],
    },
  }
  ```
- `validateEventPayload`: `["supplierName", "amount"]`
- `DEFAULT_EVENT_TEMPLATES.supplierPaymentSent`:
  - supplier: `"✅ *تم إرسال دفعة*\n... شكراً لتعاملكم 🌟"`
  - owner: `"💸 *دفعة لمورد*\n{supplierName}: {amount} ج.م ({method})..."`
- `samplePayload.supplierPaymentSent` للـ preview في الـ UI

### 2. `src/utils/automation/buildDailyReport.js`

`DEFAULT_AUTOMATION_CONFIG.eventTriggers.events.supplierPaymentSent` بـ
`enabled: false` (admin must opt-in).

### 3. `src/pages/AutomationPg.jsx`

`eventTypes` array — `"supplierPaymentSent"` مضاف بعد `"paymentReceived"`.
الـ UI بـ يـ auto-render العنصر الجديد عبر EVENT_VARIABLES (no other UI work
needed).

### 4. `api/automation-tick.js` — `scanRecentSupplierPayments`

```js
async function scanRecentSupplierPayments(db, cfg, eventCfg, cairoDate){
  // Iterates supplierPayments (last 24h)
  // Skips method=endorsed_check + any method containing "شيك"
  // Approximate supplier balance: -Σ(supplierPayments per supplier)
  // Fires processEvent with idempotencyKey `supplierPay:${id}`
}
```

Wired in main loop section **C3** (after paymentReceived/checkPaymentReceived):
```js
const supPayCfg = (eventTriggers.events || {}).supplierPaymentSent;
if (supPayCfg?.enabled) {
  const r = await scanRecentSupplierPayments(db, cfg, supPayCfg, cairo.date);
  ...
}
```

### 5. `src/pages/TreasuryPg.jsx` — Client-side instant fire

Mirror of the existing `_instantPay_needed` block:
```js
const _isCheckMethod = String(txMethod||"").indexOf("شيك") >= 0;
const _instantSupplierPay_needed = (linkedSupplierId && txType==="out"
  && txCategory==="دفعة مورد" && !editId && !_isCheckMethod);
const _instantSupplierPay_id = _instantSupplierPay_needed ? gid() : null;
const _instantSupplierPay_supplier = _instantSupplierPay_needed
  ? suppliers.find(x=>x.id===linkedSupplierId) : null;
```

The supplierPayments push uses `_instantSupplierPay_id` so the cron's
idempotencyKey matches the client's, ensuring atomic claim de-duplication
(V19.76.3 atomic claim).

The instant fire block computes balance and POSTs to `/api/event-trigger`:
```js
{
  eventType: "supplierPaymentSent",
  payload: { supplierName, amount, method, balance, date, office },
  supplierPhone,
  idempotencyKey: "supplierPay:" + _instantSupplierPay_id,
}
```

## Filtering — no overlap with checkPaymentIssued

الـ supplierPaymentSent يـ skip:
- `method === "endorsed_check"` — fired by `checkEndorsed` event
- `method` تحتوي على "شيك" — fired by `checkPaymentIssued` event

كده الـ user يبعت **رسالة واحدة فقط** لكل supplier payment — مش رسالتين.
