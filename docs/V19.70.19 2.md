# V19.70.19 — Structural workaround: `<th>` → `<td class='h'>`

**Date:** 2026-05-06
**Type:** fix (probable real fix for the Arabic PDF bug)
**Status:** shipped — pending user verification

## The user's insight

After three failed attempts at fixing the Arabic shaping bug via font loading (V19.70.14 link injection, V19.70.15 FontFace API, V19.70.16 system fonts), the user pointed out:

> "العناوين مكتوبة تمام، ليه مانعملش تجميد للعناوين المطلوبة للجدول وتكون زيها زي الكتابة النصية الثابتة حتى لو هانغير هيكل التقرير"

Translation: "The headings/titles render correctly. Why don't we freeze the table headers and make them like the static text — even if we have to change the report structure."

This was the breakthrough. The bug isn't about fonts at all — it's about which HTML element. Body `<td>` cells render Arabic correctly with the same font, same weight. `<h2>` headings render correctly. Only `<th>` breaks. So the fix is to stop using `<th>` for the column headers.

## Why this should work

html2canvas's internal iframe handles `<th>` and `<td>` via different code paths because they have different default browser styling and different interactions with table layout. The Arabic bidi/shaping pipeline breaks specifically when the element is `<th>`. Switching to `<td class='h'>` keeps the visual identical (gradient background, bold weight, right alignment) but routes through the working code path.

## Implementation

**Before:**
```html
<table>
  <tr>
    <th>العميل</th><td>محمد</td>
    <th>التليفون</th><td>0123456789</td>
  </tr>
</table>
<table>
  <thead>
    <tr><th>الموديل</th><th>الوصف</th>...</tr>
  </thead>
  ...
</table>
```

**After:**
```html
<table>
  <tr>
    <td class='h'>العميل</td><td>محمد</td>
    <td class='h'>التليفون</td><td>0123456789</td>
  </tr>
</table>
<table>
  <thead>
    <tr><td class='h'>الموديل</td><td class='h'>الوصف</td>...</tr>
  </thead>
  ...
</table>
```

CSS:

**Before:**
```css
th {
  background: linear-gradient(180deg, #E2E8F0, #CBD5E1);
  font-family: 'Tahoma', 'Arial', sans-serif;
  font-weight: 700;
  font-size: 10px;
  /* ... */
}
```

**After:**
```css
.h {
  background: linear-gradient(180deg, #E2E8F0, #CBD5E1) !important;
  font-family: 'Cairo', sans-serif;
  font-weight: 700;
  font-size: 10px;
  /* ... */
}
```

Note: switched the font-family from `'Tahoma', 'Arial', sans-serif` (V19.70.16's attempt) back to `'Cairo', sans-serif`. The body cells were always using Cairo and rendering correctly — the bug was about the element type, not the font. Cairo is the brand font, restored.

`!important` on the background: defensive against CSS specificity wars with the existing `td {}` rule.

## Scope

Only `buildOneCustomerHTML` in `CustDeliverPg.jsx` is touched. Every other `<th>` use in the codebase is for browser-native print pages (`printPage`), which works correctly because the browser handles Arabic shaping in its print pipeline. Those don't need to change and weren't broken.

## Lesson

When facing rendering bugs in html2canvas with complex scripts (Arabic, Hebrew, Devanagari):
1. First check if the bug is element-specific (different elements render differently)
2. Try swapping the broken element for a working one (with CSS to preserve visuals)
3. Only resort to refactoring the entire PDF pipeline (Approach A) if structural workarounds fail

The user's intuition that "static text works" was the right diagnostic frame — the bug was about html2canvas's element-specific handling, not about fonts.
