## الإصلاح: Atomic claim pattern

أضيفت `claimEvent(db, ...)` transaction في `api/_eventProcessor.js`. قبل أي
bridgeSend، الـ caller لازم يـ "claim" الـ idempotencyKey بـ writing entry فيها
`inFlight: true` داخل Firestore transaction. أي caller تاني يـ شوف الـ inFlight
entry ويرجع `deduped: in-flight`.

```js
// V19.76.3 logic in processEvent
if (!force) {
  const claim = await claimEvent(db, { idempotencyKey, eventType, payload, source });
  if (!claim.claimed) {
    return { ok: true, status: 200, body: { ok: true, deduped: true, reason: claim.reason } };
  }
}

// Then bridgeSend()...
// Then recordResult(success: true) — UPDATES the existing inFlight entry
```

### `claimEvent` semantics

| Existing entry | Result |
|---|---|
| لا يوجد | `claimed: true`، يكتب inFlight entry جديدة |
| `success: true` | `claimed: false, reason: "already-succeeded"` |
| `inFlight: true` و age < 60s | `claimed: false, reason: "in-flight"` |
| `inFlight: true` و age ≥ 60s | `claimed: true`، يـ reclaim (assume crash) |
| `success: false` (failed earlier) | `claimed: true`، يحاول retry |

الـ stale lock timeout = 60 ثانية — أكبر بكتير من أسوأ الـ bridge typing delay (25s).

### `recordResult` updated

قبل V19.76.3:

```js
history.unshift({...});  // دايماً يضيف entry جديد
et.eventHistory = history.slice(0, 100);
```

بعد V19.76.3:

```js
const existingIdx = history.findIndex(h => h.idempotencyKey === idempotencyKey);
if (existingIdx >= 0) {
  history[existingIdx] = finalEntry;  // يـ replace inFlight بـ final status
} else {
  history.unshift(finalEntry);  // fallback: لو claim مَ تنفذش
}
```

كده الـ history بقت entry واحدة per event بـ:
- `at` — وقت الـ claim
- `completedAt` — وقت الـ recordResult
- `success: true/false`
- `inFlight` flag مش موجودة في الـ final entry

## السيناريو الجديد (الـ race مقفولة)

```
T=0    client.fire → processEvent
       → claimEvent transaction:
         - read history → empty
         - write inFlight entry
         - return { claimed: true }
       → bridgeSend (start)
T=0.5  cron.tick → processEvent
       → claimEvent transaction:
         - read history → finds inFlight entry, age=0.5s < 60s
         - return { claimed: false, reason: "in-flight" }
       → return { deduped: true } ✅ NO BRIDGE CALL
T=2    client's bridgeSend resolves
       → recordResult: findIndex finds inFlight, replaces with success entry
       → returns { sent: 1 }
```

**الناتج:** رسالة واحدة فقط للعميل، history صحية، no orphans.

## Edge cases handled

1. **Force mode** (manual replay from Automation tab) — `force: true` يـ bypass
   the claim بحيث الأدمن يقدر يعيد الـ trigger يدوياً حتى لو inFlight موجود.
2. **Bridge fails** — `recordResult(success: false)` يحدّث الـ inFlight entry بـ
   `success: false`. الـ next caller يـ pass الـ `claimEvent` check (entry غير
   ناجح + إن كان stale lock فسيـ reclaim).
3. **Stale lock** (Vercel function crashed) — الـ next caller بعد 60s يـ reclaim
   و يحاول مرة جديدة.
4. **Concurrent claims في نفس الـ instant** — Firestore transactions تضمن
   serializable ordering. الواحد بس بـ يـ claim؛ الباقي يـ see the inFlight.
