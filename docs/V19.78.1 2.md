## السياق

User report:
> "العميل يبعت 'السلام عليكم'، 'ازيك ياغالي'، 'ايه الاخبار' — في الطبيعي الـ Agent
> بيرد على كل رسالة لوحدها نفس الرد. عاوز رد واحد. هل نقدر نظبط ده؟"

Without debouncing, every webhook fires the orchestrator immediately. Three
quick greetings = three LLM calls = three replies. Customer gets spam, the
factory pays 3× the cost.

V19.78.1 ships a **trailing-debounce** mechanism that batches rapid messages
into one LLM turn.

---

## How it works

### "Latest caller wins" pattern

Every webhook that arrives:
1. Pushes its message to a Redis list `mqueue:{phone}` with a monotonic seq
2. Sets `mlatest:{phone}` = its seq (atomic INCR)
3. Sleeps `TEXT_DEBOUNCE_MS` (default 4000ms)
4. Checks: am I still `mlatest`? If yes → flush the queue and process.
   If no → drop out silently (a newer caller will handle it).

This pure-Redis approach has no race conditions even with 10 concurrent
webhooks — only one will end up the "latest" and process the entire batch.

### Combined batch format

The orchestrator receives a single user message that frames the batch:

```
[العميل بعت 3 رسائل متتالية في خلال ثوانٍ — رد على المضمون كله مرة واحدة]

1. السلام عليكم
2. ازيك ياغالي
3. ايه الاخبار
```

Claude understands the framing and replies once with a unified response.
Single-message turns (most common) skip the framing and pass the text as-is.

### Configurable

Env var `TEXT_DEBOUNCE_MS`:
- **0** — disable batching (every webhook → immediate orchestrator call). Restores pre-V1.0.4 behavior.
- **2000-3000** — aggressive batching (good for chatty customers).
- **4000** (default) — balanced — covers typical typing-pause flurries.
- **6000-8000** — relaxed (more batching, more delay before reply).
- **Max 15000** — clamped at validation time.

To change without redeploy:
```bash
ssh root@77.237.235.160
cd /root/clark-ai-agent
# add or edit:
echo 'TEXT_DEBOUNCE_MS=3000' >> .env
pm2 restart clark-ai-agent
```

### What's NOT debounced

- **Voice messages** — bypass the queue. Each voice note is a standalone
  intent (transcribe + reply immediately).
- **Image messages** — same. The customer sent the image deliberately;
  delaying it would feel broken.
- **Tool result loops** — the inner LLM round-trips for tool execution
  are inside the orchestrator and not affected.

## Test scenarios

1. **Single-message** — اكتب "احكي بياناتي" → الـ Agent يرد بعد 4s (default debounce). Latency tradeoff acceptable for typical conversations.

2. **3-message flurry** — اكتب الـ 3 رسائل دي بسرعة:
   ```
   السلام عليكم
   ازيك ياغالي
   ايه الاخبار
   ```
   متوقّع: الـ Agent يرد **مرة واحدة** بـ greeting موحّد.

3. **Mixed flurry + question** —:
   ```
   السلام عليكم
   كم باقي عليّ؟
   ```
   متوقّع: رد واحد فيه greeting + الرصيد.

4. **بعد فترة** — اكتب رسالة، استنى 10 ثواني، اكتب تانية:
   متوقّع: ردين منفصلين (الـ window 4s انتهى).

5. **Tune for chatty customer** — `TEXT_DEBOUNCE_MS=6000` للعملاء اللي بيكتبوا ببطء، `TEXT_DEBOUNCE_MS=2000` للسريعين.

6. **Disable** — `TEXT_DEBOUNCE_MS=0` لو عاوز immediate replies (للـ debugging).

## Why this is the right approach

**Alternative considered:** "First caller wins" with a sliding window — process the first message immediately, queue the rest. **Rejected** because the first message is often "السلام عليكم" — the LEAST informative. We want the agent to see the context (the actual question) before replying.

**Alternative considered:** Smart batching based on punctuation — wait if the message ends with no terminal punctuation. **Rejected** as too brittle for Arabic + emoji-heavy WhatsApp conversations.

**Final design:** Trailing debounce, latest caller wins. Mirrors how
human assistants behave — wait until the customer pauses, then read the
whole context, then reply.
