# V19.70.24 — TTF 403 fix + delete customer + audit width + PDF default OFF

**Date:** 2026-05-06
**Type:** fix, ux
**Status:** shipped

## What landed

Four issues from user feedback:

1. **Cairo TTF 403** — V19.70.23's CDN URL returned 403, breaking PDF generation entirely
2. **PDF default OFF** — user wants explicit opt-in, not opt-out
3. **Delete customer from matrix** — currently only possible when row has values
4. **Audit popup too wide** — should fit content based on customer count

## 1. Cairo TTF 403 fix

V19.70.23 fetched Cairo TTFs from `cdn.jsdelivr.net/gh/google/fonts@main/...` — jsdelivr's GitHub passthrough. That endpoint is now returning 403 for these paths (Google Fonts repo doesn't permit raw passthrough anymore).

**Fix:** switched to fontsource's npm package, which IS hosted on jsdelivr. The same package we used in V19.70.15 for the woff2 files (which worked). Their `files/` directory contains TTF too:

```
https://cdn.jsdelivr.net/npm/@fontsource/cairo@5.0.13/files/cairo-arabic-400-normal.ttf
https://cdn.jsdelivr.net/npm/@fontsource/cairo@5.0.13/files/cairo-arabic-700-normal.ttf
```

The Arabic subset is small (~30-50KB per weight) — much smaller than full Cairo TTF (~250KB).

**Defensive layer:** added a fallback chain. If jsdelivr's `cairo@5.0.13` fails (CDN outage, package unpublished), try `unpkg.com`, then `cairo@4.5.13` as a last-resort older version.

```js
const CAIRO_REGULAR_TTF_URLS = [
  "https://cdn.jsdelivr.net/npm/@fontsource/cairo@5.0.13/files/cairo-arabic-400-normal.ttf",
  "https://unpkg.com/@fontsource/cairo@5.0.13/files/cairo-arabic-400-normal.ttf",
  "https://cdn.jsdelivr.net/npm/@fontsource/cairo@4.5.13/files/cairo-arabic-400-normal.ttf",
];
```

The `_fetchAsBase64` helper in `arabicPdf.js` accepts an array now and tries each URL in order, falling back on any failure (network, HTTP error, CORS rejection). Throws only when ALL fail.

## 2. PDF default OFF

V19.70.23 flipped the `📎 إرفاق نسخة PDF` checkbox to default ON, assuming the new pipeline was working. The user prefers it default OFF — they want to opt in explicitly when they need a PDF, since text-only is faster and works for most batches.

The checkbox stays. The default reverted to OFF. The subtitle reflects the new default: "رسالة تفاصيل نصية فقط (افتراضي — أسرع وأخف)".

## 3. Delete customer from matrix

The existing 🗑 delete button was nested inside the per-customer actions column (`<td>` containing all the action buttons like 📱 WhatsApp, 🏷 label, 🗑 delete). That whole column was conditionally rendered with `{rowTotal > 0 && ...}` — meaning if a customer had no quantities entered, the delete button was hidden along with the others.

Result: the user couldn't delete an empty customer (one they added by mistake) from inside the matrix.

**Fix:** added a small ✕ icon next to the customer's name in the first (sticky) column. Always visible when `sessCanEdit`. Independent of `rowTotal`. Same safety check (block + show 🔒 if there are committed sales for this customer in this session).

The icon is 11px, red (`#EF4444`), with a soft red background tint so it's distinguishable from the customer name without being intrusive. Clicking opens the existing confirm prompt; on confirm, removes the customer from `custIds`, clears matching cells from both the persisted grid AND the localGrid (so the row disappears immediately, no transient flash).

## 4. Audit popup width fit-content

Before: `width: 100%; maxWidth: window.innerWidth - 48` — full screen regardless of content.

After: `width: fit-content; maxWidth: Math.min(viewport - 48, 240 + visCusts.length * 95 + 200)`.

The 95mm-per-customer-column heuristic mirrors the matrix popup's sizing. With 3 customers, the popup is ~525mm wide. With 15 customers, it caps near the viewport width.

`minWidth` set to 480px on desktop so the header buttons (تحليل / تغيير العملاء / حفظ / إغلاق) don't wrap awkwardly when there are very few customers.

Mobile keeps the existing full-width behavior.
