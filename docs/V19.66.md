## الإصلاحات

### 🛡️ Double-submit guard على confirmSale

**File:** `src/pages/CustDeliverPg.jsx:72, 2819-2828`

قبل V19.66 الـconfirm-sale popup كان عنده guard `total<=0` فقط. ضربتين سريعتين على "تأكيد البيع":
1. كل واحد يعدي الـvalidation
2. كل واحد يـcall `updOrder(o => o.customerDeliveries.push(entry))`
3. الـjournal idempotency عبر `entry._key` يحمي الـaccounting (واحد فقط ينعمل post)
4. لكن الـ`customerDeliveries` array بيتضمن duplicate → الـqty يتضاعف في الـorder

دلوقتي `qrSaleSubmittingRef = useRef(false)`:
- أول call يـset الـref true
- `setTimeout(()=>{ref.current=false}, 800)` — release بعد 800ms
- أي click ثاني خلال الـwindow → `if(qrSaleSubmittingRef.current) return`

### 💰 Sale-return price ميـvعود لـlist price

**File:** `src/utils/accounting/postingRules.js:105` + `src/pages/CustDeliverPg.jsx:2944-2947`

قبل V19.66:
- `buildSaleReturnEntry` يستخدم `Number(order.sellPrice)` فقط للـnet calculation
- لو الـoriginal sale كان بـcustom price (e.g. خصم خاص لعميل)، الـreturn يـreverse عبر الـlist price → AR credit بالـlist price مش الـactual price المباع → debit drift دائم

V19.66:
1. **في buildSaleReturnEntry**: `const price = Number(ret.price) || Number(order.sellPrice) || 0;`
2. **في CustDeliverPg confirmSale return path**: نـlookup الـmatching delivery (بـsessionId+custId)، لو discounted نـcopy الـ`price` على الـreturn entry
3. الـjournal دلوقتي يـreverse الصح حتى للـdiscounted sales

### 🛡️ Free-sale `_key` collision

**File:** `src/pages/CustDeliverPg.jsx:2882`

قبل V19.66:
- `entry._key = oid + ":saleDelivery:" + sessId + ":" + custId + ":" + date`
- بيع نفس العميل مرتين من نفس الـlinked-session في نفس اليوم = identical `_key`
- `autoPost.sale` يـuse الـ_key كـsourceId لـidempotency → الـsecond sale يتـskip silently
- الكمية تتسجل في `customerDeliveries` لكن مش في الـjournal = **silent over-sale**

V19.66: ضفنا `Date.now()` ms timestamp في الـ_key:
```js
entry._key = oid + ":saleDelivery:" + sessId + ":" + custId + ":" + date + ":" + Date.now();
```

كل sale unique. الـlegitimate re-sale (e.g. عميل اتسلم 50 الصبح، رجع و اتسلم 30 العصر) يـpost الجورنال الصح للاتنين.

### 💰 Float drift في invoices.js — كل math rounded

**File:** `src/utils/invoices.js`

قبل V19.66 الـsubtotal/discount/total كانوا يـaccumulate floating-point error:
```
1234 * 0.005 = 6.17 → stored: 6.170000000000001
existing.subtotal += 6.170000000000001 → eventual drift
```

8 موضع في invoices.js (sales build، sales upsert merge، sales upsert create، purchase build، purchase upsert merge، purchase upsert create) ما كانش فيهم `r2()` wrap. دلوقتي:
- Imported `r2` from `format.js` (`Math.round(n*100)/100`)
- كل multiplication/subtraction/percentage routed through r2

نفس الـpattern المستخدم بالفعل في `postingRules.js` (الـjournal entries كانت rounded).

## 📝 خطة الاختبار

### Critical paths

1. **Double-submit**: افتح QR sale popup، اكتب qty، اضغط "تأكيد البيع" مرتين سريع (double-tap) → بيع واحد بس يحصل (الـcustomerDeliveries فيها entry واحد، مش اتنين)

2. **Discounted return**:
   - بيع موديل لعميل بسعر مخفّض (custom price = 100 بدلاً من list 150)
   - أعمل return للـsame qty
   - افتح الـjournal — لازم credit AR بـ100×qty (مش 150×qty)
   - الـcustomer balance يـreturn لـzero (مش debit drift)

3. **Free-sale re-sale**:
   - بيع 30 قطعة من سيشن A
   - بيع 20 قطعة تانية من نفس السيشن، نفس العميل، نفس اليوم
   - افتح الـjournal — لازم 2 entries (اتنين منفصلين، مش واحد)
   - الـcustomerDeliveries فيها 2 entries بـ`_key`s مختلفة

4. **Float drift**:
   - أنشئ فاتورة بـ3 line items، unitPrice = 33.333، qty = 7 لكل واحد
   - الـsubtotal لازم يكون 700.00 (مش 700.0000000003)
   - أضف خصم 17.5% — الـtotal منطقي بدون عشرات-أصفار

### Regression

5. كل الـsale flows (free، linked، override، quantity adjustments) لسه شغّالة كما قبل
6. الـauto-post journal entries للـsales لسه يحدثوا
7. invoice draft consolidation (V18.65) لسه يـmerge same-day same-customer drafts
