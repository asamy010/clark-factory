## Issue #1: تأكيد الاستلام — Series and broken qty stacked

### السياق

User request:
> "في شاشة الاستلام لمخزن الجاهز عاوز يظهر الكميات السيري والكميات الكسر فوق
> بعض والتأكيد نفس الكلام عشان يكون واضح الفرق دايمأ"

شاشة `تأكيد استلام المخزن` كانت بـ تعرض كل قيمة في رقم واحد بدون تفريق بين
الـ سيري (📦 series) والـ كسر (🧩 broken). الفرق ده مهم لأن:
- الـ سيري = pieces grouped في rack (e.g. rackSize=4 → 1 series = 4 قطع)
- الـ كسر = loose pieces

### الإصلاح

كل خلية في 3 أعمدة (معلّق، تسليم مخزن جاهز، الفرق) دلوقتي بـ تعرض صفين:
- **📦 سيري** (أخضر `#10B981`) فوق
- **🧩 كسر** (بنفسجي `#8B5CF6`) تحت

**معلّق column** بـ يـ split الـ pending deliveries:
```js
// V19.76.7: filter o.deliveries by d.type
const isBroken = d.type === "broken";
pendingMap[key].pendingSeriesQty += (isBroken ? 0 : qty);
pendingMap[key].pendingBrokenQty += (isBroken ? qty : 0);
pendingMap[key].pendingSeriesIdxs.push(di);  // OR pendingBrokenIdxs
```
Legacy entries بدون `type` بـ تـ default لـ "series" (matches `getConfirmedBrokenStock`).

**تسليم مخزن جاهز column** بقى inputين منفصلين بدل واحد:
- `rcvItems[orderId] = { series: number, broken: number }` (was just `number`)
- Helpers `itemSeries(v)`, `itemBroken(v)`, `itemQty(v)` للـ backward compat

**Scan + manual add routing** بقى يحدد الـ type تلقائياً من الـ scanMode toggle:
- `scanMode === "series"` → +rackSize للـ series slot
- `scanMode === "piece"` → +1 للـ broken slot

**Confirm logic** بـ يعالج كل النوعين separately:
```js
const consume = (idxs, need) => { /* set status confirmed, decrement remaining */ };
consume(p.pendingSeriesIdxs, qtySeries);
consume(p.pendingBrokenIdxs, qtyBroken);
```
كده الـ `type` metadata بـ يفضل مع الـ delivery بعد التأكيد، فـ `getConfirmedSeriesStock`
و `getConfirmedBrokenStock` بـ ترجع أرقام صحيحة.

**الفرق column** بـ يحسب diff لكل نوع منفصل: `diffSeries = valSeries - pendingSeriesQty`.

**عمود مباع للعميل** سايب flat — `customerDeliveries` data ما تحملش الـ
type الحالي (per V18.21 السلز بياخدوا من الـ series stock أولاً).

## Issue #2: Order details cost table reordering

### السياق

User report:
> "في تفاصيل الاوردر لما اضيف بند اضافي تكلفة يظهر قبل الاجمالي والاجمالي في
> الاخر، عشان الرقم لتكلفة القطعة مش صح، عشان جمع الاجمالي على تكلفة الاتشغيل"

ترتيب جدول `ملخص تكلفة الموديل` كان غلط من ناحية الـ UX:
1. تكلفة الخامات
2. تكاليف الاكسسوار
3. **الإجمالي (sub-total)** ← في النص! ما بيشملش الإضافات
4. هالك تسوية (لو في)
5. تكلفة إضافية / تشغيل
6. **التكلفة الفعلية** (grand total)

الـ user كان بيشوف الـ "الإجمالي" قبل الإضافات → يحس إن الـ total مش شامل
كل التكاليف. كمان الـ "تكلفة القطعة" للـ التكلفة الفعلية كانت محسوبة بـ
`/ deliveredQty` (e.g. 48) بدل `/ cutQty` (e.g. 400)، فطلع 3361 ج.م/قطعة بدل
403 — رقم مش consistent مع باقي الصفوف.

### الإصلاح

ترتيب جديد:
1. تكلفة الخامات
2. تكاليف الاكسسوار
3. هالك تسوية (لو في)
4. تكلفة إضافية / تشغيل
5. **الإجمالي** ← الصف الأخير، شامل كل اللي فوق

```jsx
// V19.76.7: extras + settlement BEFORE the total. Total = LAST row.
const totalAllPer = cutQty > 0 ? totalAllCost / cutQty : 0;
return <table>...
  <tr>الخامات</tr>
  <tr>الاكسسوار</tr>
  {settlement && <tr>هالك تسوية</tr>}
  {extraCosts.map(x => <tr key={x.id}>{x.category}</tr>)}
  <tr style={accent}>
    <td>الاجمالي</td>
    <td>{totalAllCost}</td>
    <td>{cutQty > 0 ? totalAllPer : "—"}</td>
  </tr>
</table>;
```

الـ per-piece للـ grand total دلوقتي = `totalAllCost / cutQty` — نفس الـ
denominator اللي بـ يستخدمه باقي الصفوف، فالعمود consistent.

### مثال في الـ user's screenshot

| البند | قبل V19.76.7 | بعد V19.76.7 |

| تكلفة الخامات | 95,090 / 237.73 | 95,090 / 237.73 |
| تكاليف الاكسسوار | 18,400 / 46 | 18,400 / 46 |
| الاجمالي (sub-total) | 133,290 / 334 ❌ في النص | (محذوف) |
| تشغيل من القص | 28,000 / 70 | 28,000 / 70 |

| **الإجمالي** | — | **161,290 / 403 ج.م/قطعة** ✓ في الآخر |
