# V19.70.14 — First attempt at Arabic PDF glyph shaping (didn't fully fix)

**Date:** 2026-05-06
**Type:** fix (incomplete)
**Status:** shipped, but the bug persisted — see V19.70.15 for the real fix

## The bug

User report: عناوين الجداول في الـPDF العربي ملخبط ومش مقروء وبالعكس. Screenshots showed:
- `العميل` rendered as `لـعميل`
- `التاريخ` rendered as `لـتايخ`
- `التليفون` rendered as `لـتيفون`

Customer name + address (in `<td>` cells with `font-weight: 400`) rendered correctly. The breakage was confined to `<th>` headers.

## Root cause analysis (V19.70.14's hypothesis)

`<th>` in the CSS uses `font-weight: 800` (Cairo Bold). The offscreen html2canvas capture ran before the Cairo Bold variant was loaded into the browser's font cache. The browser fell back to Arial / system fonts which don't shape Arabic ligatures correctly — every letter rendered in its isolated form instead of connecting with neighbors.

## What V19.70.14 attempted

`ensureCairoLoaded()` helper added to `htmlToPdf.js`:

1. Inject `<link>` to Google Fonts CSS for Cairo with all weights (400, 500, 600, 700, 800, 900) if not already present.
2. Call `document.fonts.load("400 12px Cairo")`, `document.fonts.load("700 12px Cairo")`, `document.fonts.load("800 12px Cairo")`, `document.fonts.load("900 12px Cairo")` in parallel.
3. `await document.fonts.ready` as a fallback.
4. 200ms extra delay for font metrics to settle.

Plus defensive measures on the offscreen container:
- `setAttribute('dir', 'rtl')` and `setAttribute('lang', 'ar')` as HTML attributes (not just CSS).
- `letterRendering: true` in html2canvas options.
- Wait time bumped from 100ms to 250ms before capture.

## Why it didn't work

User confirmed the headers were still scrambled.

The hypothesis was correct (font race condition), but the **mechanism wasn't strong enough**. `document.fonts.load(string)` is a *trigger*, not a *wait*. The API tells the browser "I will use this font, please load it" and returns a promise — but the promise can resolve while the actual TTF binary is still in flight. On a fresh page-load with cold cache, the await would resolve in ~5-10ms while the binary was still downloading from Google Fonts.

## What worked: V19.70.15

Replace `document.fonts.load(string)` with explicit `new FontFace(...).load()` backed by direct WOFF2 binaries from fontsource CDN. `face.load()` resolves only after the binary has finished downloading and parsing — no race condition possible.

See `docs/V19.70.15.md` for the real fix.
