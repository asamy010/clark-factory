## 🧩 ميزة المحافظ — الحالة التفصيلية الكاملة (بعد هذا الـ session)

**نموذج البيانات** (على account object في `cfg.treasuryAccounts`):
```js
{ id, name, ownerEmail, type:"wallet",
  walletNumber, icon, image,
  balanceCap, monthlyWithdrawCap,        // حدود (افتراضي 200000)
  tiers:[{from, to, pct, min, max}] }    // عمولة %؛ to=null=مفتوح؛ min/max=clamp اختياري (0=بدون حد)؛ legacy fee=ثابت
```

**مصفوفة الاكتمال:**
| | إنشاء | تعديل | حذف | تحويل |
|---|:---:|:---:|:---:|:---:|
| **العمولة** | ✅ (V204) | ✅ V213 | ✅ V211 | ⛔ **بقرار أحمد (بدون)** |
| **الحدود** | ✅ (V205) | ✅ V212 | — | ✅ V214 |
| **min/max للعمولة** | ✅ V215 لكل الشرائح | | | |

**الدوال المفتاحية (TreasuryPg.jsx):**
- `computeWalletFee(w, amount)` — نسبة % من الشريحة المطابقة، ثم **clamp بين min/max** (0=بدون حد، متوافق رجعياً). fallback لـ legacy fixed `fee`.
- `walletMonthOut` (useMemo) — مجموع كل حركات الـ `out` على المحفظة في الشهر الحالي. **بيشمل legs التحويل والعمولة تلقائياً** (لأنهم كلهم out على الحساب).
- `saveTxWithLimits` — **pre-flight wrapper** قبل `saveTx`. فرع `if(!editId)` للإضافة (الأصلي، اتساب كما هو) + فرع `else` للتعديل (V212: recompute-excluding-self + حارس الزيادة `_post>_pre`).
- **العمولة على التعديل (V213):** في فرع التعديل بـ `saveTx` — **شيل-وأعِد-إنشاء** لابن العمولة (`walletFeeFor===editId`). حارس `!tx.walletFeeFor` يمنع توليد عمولة على صف عمولة. القيد القديم بيتعكس (`_walletFeeOldForReverse`)، الجديد بيتنشر عبر `_newFeeEntry` autoPost الموجود.
- **حذف العمولة (V211):** `delTx` + `bulkDeleteTxs` — الفلتر بيشيل `t.walletFeeFor===id` + `autoPost.reverse` لقيد العمولة (فردي وجماعي).
- **حدود التحويل (V214):** `transferCapBlock(from,to,amt)` (يفحص سحب-شهري على المصدر + رصيد على المستقبِل) + `submitTransferWithLimits` / `approveTransferWithLimits` (**pre-flight wrappers**؛ `submitTransfer`/`approveTransfer` الأصليين اتسابوا حرف بحرف).

**الـ commission preview** في فورم الحركة بقى edit-aware (V213) + بيختفي على صف العمولة نفسه.

### ⚠️ قرار مهم موثّق (متعملوش تاني):
**التحويلات بين الخزن = بدون عمولة** (قرار أحمد، V214). السبب: التحويل حركة داخلية بين خزن الشركة مش سحب فعلي. ده بيحافظ على نموذج التحويل النضيف (2 legs + transferId) ومنطق الحذف بالـ transferId. **متقترحش عمولة على التحويلات تاني** إلا لو أحمد طلب صراحةً.

## 🛡️ دروس مهمة من هذا الـ session

1. **فحص توازن الأقواس بدون build env** (الطريقة اللي اشتغلت): قارن عدد `{}`/`()`/`[]` في الملف الحالي مع نسخة HEAD عبر `python3` — الفرق المفروض يكون متساوي (نفس عدد الفتح والقفل)، والـ net للملف ككل يطابق HEAD (اللي بيـ build على Vercel). بيمسك أي خطأ قوسي من تعديل فوراً.
   ```python
   cur=open(f).read(); head=git_show("HEAD:"+f)
   for o,c in [("{","}"),("(",")"),("[","]")]:
       assert cur.count(o)-head.count(o)==cur.count(c)-head.count(c)
   ```
2. **نمط الـ pre-flight wrapper للخزينة:** عشان تضيف فحص/gate على دالة خزينة مُتحقَّقة بدون ما تلمسها — اعمل wrapper async يعمل الفحص (`await ask`) ثم يفوّض للدالة الأصلية، ووصّل الزر/الـ confirm بالـ wrapper. النمط ده (`saveTxWithLimits`→`saveTx`، `submitTransferWithLimits`→`submitTransfer`، `approveTransferWithLimits`→`approveTransfer`) = صفر regression على المنطق الأصلي.
3. **رابط العمولة `walletFeeFor`:** entry العمولة ("عمولة محفظة") = حركة out عادية على نفس المحفظة، مربوطة بالأب عبر `walletFeeFor:parentId`. **كل دورة الحياة لازم تزامنه:** الإنشاء (V204)، التعديل (V213 شيل-وأعِد-إنشاء)، الحذف (V211 cascade). أي feature جديد يلمس الحركات لازم ياخد باله من الرابط ده.
4. **`walletMonthOut` بيشمل التحويلات والعمولة أصلاً** — فالعرض على بطاقة المحفظة والحسبة كانوا بيعكسوا التحويلات؛ اللي كان ناقص بس هو **التطبيق وقت الإنشاء** (V214).
5. **أحمد بيـ push أثناء الـ session** (مش بس في الآخر) — فممكن تلاقي origin اتقدّم. اتأكد من `git rev-list --left-right --count origin/main...HEAD` قبل ما تقول حاجة عن الـ unpushed.
