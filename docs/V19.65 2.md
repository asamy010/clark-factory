## 🚨 CRITICAL: enforceDataLimits silent-delete bug

### الـbug

في `src/utils/dataLimits.js`، الـenforceDataLimits بتـtruncate arrays لـ`next.treasury`/`hrLog`/`custPayments`/etc. لو طولها أكبر من الـlimit (3000-5000). لكن:

- بعد V16.74 (split migration)، الـ`next.treasury` بيكون الـ**full merged array** (الـhydration في `upConfig` بتـcopy splitData كاملاً قبل ما `fn(next)` يـrun)
- الـenforceDataLimits بتـtruncate لـ3000
- الـ`syncAllSplitChanges(splitBefore, splitAfter)` بيشوف الـIDs الناقصة في `splitAfter` → بيعمل `deleteDoc()` على day docs اللي تحتوي على entries مفقودة
- النتيجة: **مئات/آلاف من treasury entries (سنين من الـhistory) تتمسح بصمت كل مرة الـlimit يتعدّى**

ده active منذ V16.74. لو الـuser ميـلاحظش "treasury history مفقود قبل تاريخ كذا"، الـbug فضل invisible.

### الـFix

`enforceDataLimits` دلوقتي تـcheck الـsplit migration flags وتـskip الـmigrated fields:

```js
function _migratedFields(d){
  const set = new Set();
  if (d[SPLIT_FLAG_V1674]) SPLIT_FIELDS_V1674.forEach(f => set.add(f));
  if (d[SPLIT_FLAG_V1949]) SPLIT_FIELDS_V1949.forEach(f => set.add(f));
  if (d[SPLIT_FLAG_V1950]) SPLIT_FIELDS_V1950.forEach(f => set.add(f));
  if (d[SPLIT_FLAG_V1952]) SPLIT_FIELDS_V1952.forEach(f => set.add(f));
  if (d[SPLIT_FLAG_V1953]) SPLIT_FIELDS_V1953.forEach(f => set.add(f));
  return set;
}

export function enforceDataLimits(d){
  if(!d)return;
  const skip = _migratedFields(d);
  Object.entries(LIMITS).forEach(([key,limit])=>{
    if (skip.has(key)) return;/* skip migrated fields */
    /* ... existing truncation ... */
  });
}
```

الـday-docs ميـحتاجوش الـ1MB cap (each day-doc has hundreds of entries max).

## 🛡️ Other fixes

### JSON.stringify → _deepEqual في 7 places

الـpending-write cleanup logic كانت تستخدم `JSON.stringify(a)===JSON.stringify(b)`. ده order-dependent — Firestore يقدر يرجع keys بـorder مختلف → identical objects تظهر unequal → pending entries تـ-persist 30 sec → الـ`flatten()` يستخدم pending data بدل server data → users يـشوفوا stale optimistic data بـoverlay على real server data.

دلوقتي بيستخدم `_deepEqual` (export-ed من `splitCollections.js`) في:
- 4 listener pending-cleanup checks (line 2574، 2666، 2740، 2844)
- 4 registerPending* change-detection (line 2383، 2407، 2428، 2449)

### upSales + upTasks wipe guards

V19.62/63 ضافوا الـ`≥5→0` wipe guard في `upConfig`. `upSales` و `upTasks` ما عندهمش guard. لو bug زي V19.62 ظهر في الـsales/tasks write path، الـsame data destruction يحصل (wipe state + delete day-docs via syncAllSalesSplitChanges).

دلوقتي:
- `upSales` يـcheck SALES_SPLIT_FIELDS (packages، custDeliverySessions)
- `upTasks` يـcheck TASKS_SPLIT_FIELDS (tasks، stickyNotes، inventoryAudits)

### validateBeforeWrite — 4 fields added

الـlist في `dataIntegrity.js` كانت ميـcoverش empDebts، generalProducts، productCategories، hrWeeks. ضفناهم — extra layer with V19.62/63 guards.
