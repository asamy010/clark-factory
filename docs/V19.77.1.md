## What shipped

### Backend (clark-ai-agent v1.0.2-phase2.5, deployed via deploy.mjs)

**6 new tool files** in `src/tools/`:

| Tool | Description | PII Posture |
|---|---|---|
| `getCustomerInfo` | Sender's full record OR limited info on others | sender = full, others = name/type only |
| `getCustomerBalance` | Mirrors كشف الحساب formula (gross − disc − cash − checks) | sender ONLY — refuses other customer_id |
| `getCustomerOrders` | Last 30 orders by activity in active season | sender ONLY |
| `getOrderStatus` | One order by id/modelNo with sender's deliveries | sender ONLY (returns "no delivery" if not theirs) |
| `notifySalesTeam` | Pings sales team for order intent + logs `aiAgentSalesNotifications` | always sends from sender's profile |
| `notifyAdminPhoneRequest` | Bridges LID → phone mapping; logs `aiAgentSuggestions` (kind: lid_phone_mapping) | sender's wid only |

**configLoader extended** with split-collection-aware helpers:
- `loadCustPayments()` — reads `custPaymentsDays/` post-migration, falls back to inline
- `loadChecks()` — reads `checksDays/` post-migration, falls back to inline
- `loadActiveOrders()` — reads `seasons/{active}/orders/` with per-season cache (30s TTL)
- `invalidateAllCaches()` — purges everything

The new tools share the cached config so the per-message overhead is tiny
(orders cached for 30s, customers/payments/checks all cached).

**Tool registry** (`src/tools/index.js`) wired all 6 imports + REGISTRY entries.
Test count: 63 → 63 (`tools.test.js` updated to assert 11 tools).

### CLARK frontend (Tools tab)

The Tools tab in `AIAgentPg.jsx` was already forward-looking with placeholder
slots. Updates:
- Renamed `get_order_details` → `get_order_status` (matches actual tool name)
- Added `get_product_details` + `get_company_info` rows (forgot in V19.72)
- Added `deployed: true|false` flag per item
- Per-row badge: ✓ مفعّل (green) for deployed, 🚧 قريباً (indigo) for placeholder

### PII guardrails (critical)

All four customer-facing data tools refuse to expose another customer's data:

```js
if (input.customer_id && input.customer_id !== sender.id) {
  return { error: "not_self", message: "متاح فقط لصاحب الحساب." };
}
```

`get_order_status` goes further: if the sender hasn't received any deliveries
from the order, it returns `found: false` rather than leak who DID receive
them. This means the agent literally CAN'T accidentally tell customer A
"customer B got 50 pieces" — the data isn't reachable through this tool.

## Test scenarios for the user to try

1. **"احكي بياناتي"** — agent uses `get_customer_info` (no args) → returns full sender profile.
2. **"كم باقي عليّ؟"** — agent uses `get_customer_balance` → exact match with kashf in CLARK.
3. **"إيه طلباتي عند المصنع؟"** — agent uses `get_customer_orders` → list of last 30 orders.
4. **"ما حالة طلبي 3262111؟"** — agent uses `get_order_status` → details for that order if sender ordered it.
5. **"عاوز أطلب 100 قطعة من 3262111"** — agent uses `notify_sales_team` → admin gets WhatsApp + log in dashboard.
6. **"أنا أحمد المالك"** (from an LID) — agent uses `notify_admin_phone_request` → admin can link in CLARK.

PII tests — these MUST be refused by the agent:
- "كم رصيد العميل أحمد محمد؟" (asking about a different customer's balance)
- "احكي طلبات العميل 0100123456" (querying another phone)
