## ✅ الحل في V19.50

### 1. Daily-split لـ3 arrays إضافية

```
قبل V19.50:                         بعد V19.50:
─────────────                        ─────────────
factory/config = ~441 KB             factory/config = ~120 KB ✅

  purchaseInvoices: [..] ⚠️           _splitDaysV1950Done: true
  purchaseOrders: [114]   ⚠️ 57KB
  coa_backup_pre_upgrade_*  ⚠️ 24KB

                                     salesInvoicesDays/2026-05-04 = {entries:[...]}
                                     purchaseInvoicesDays/2026-05-04 = {entries:[...]}
                                     purchaseOrdersDays/2026-05-04 = {entries:[...]}

                                     backups/coa-rescued-from-config-* (×2)
```

### 2. Cleanup للـcoa_backup_pre_upgrade_*

- كل key في factory/config يبدأ بـ`coa_backup_pre_upgrade_`:
  - يُحفظ كـdoc منفصل في `backups/coa-rescued-from-config-{ts}` (مع label, timestamp, original key)
  - يُحذف من factory/config
- النتيجة: تنظيف 24 KB في الحالة الحالية.

### 3. إصلاح المصدر — `ChartOfAccountsTab.jsx`

قبل V19.50: عند ترقية شجرة الحسابات، الـbackup كان بيدمب كـkey في factory/config:
```js
d[`coa_backup_pre_upgrade_${ts}`] = JSON.parse(JSON.stringify(d.coa));  // ← bad
```

بعد V19.50: الـbackup بيتكتب في `backups/coa-pre-upgrade-{ts}` document منفصل، وأي فشل في الـbackup يلغي الـupgrade تلقائياً:
```js
await fsSetDoc(fsDoc(db, "backups", "coa-pre-upgrade-" + ts), {
  label: "CoA backup قبل ترقية شجرة الحسابات",
  coa: JSON.parse(JSON.stringify(coa || [])),
  // ... metadata
});
// ثم تعمل الترقية
```

## ⚙️ سلوك الـMigration

### السيناريو الطبيعي

1. المستخدم يفتح التطبيق على V19.50 لأول مرة
2. `_splitDaysV1949Done=true` (من V19.49 المسبق) ، `_splitDaysV1950Done=undefined`
3. listeners بقت تشتغل على 10 collections (3 V16.74 + 4 V19.49 + 3 V19.50). الـ3 الجديدة فاضية مبدئياً.
4. config useMemo: V16.74 + V19.49 fields من splitData، V19.50 fields لسه من configDoc (لأن الـflag مش on)
5. Migration V19.50 يلاقي 3 arrays في configDoc → يكتب backup كامل لـconfig → يـrescue الـcoa_backup_* لـcollection backups → ينقل الـ3 arrays لـday docs
6. Migration يحط الـflag الجديد + يحذف الحقول من configDoc + يحذف الـcoa_backup_* keys (transaction atomic)
7. configDoc snapshot يصل من Firestore بدون الحقول الـ3 ولا الـcoa_backup_*
8. config useMemo بقى يعرض V19.50 fields من splitData (الـoverride تشتغل دلوقتي)
9. كل الصفحات (SalesInvoicesPg, PurchaseInvoicesPg, PurchasePg) شغّالة عادي — **مفيش تعديل عليها**

### حالات حافة

| الحالة | السلوك |
|---|---|
| Migration فشل | الـflag مش بيتكتب → في الـrefresh الجاي يحاول تاني |
| backup فشل | الـmigration كله بيتلغي (الـbackup أول خطوة) |
| coa-rescue فشل لـkey واحد | warning في console، الـmigration يكمل (الـbackup الأساسي عنده النسخة كاملة) |
| User يسجل فاتورة أثناء الـmigration | UI مقفول بـmodal، الكتابة impossible |
| 2 جهاز شغّالين V19.50 وV19.49 | selective stripping يحمي من data loss، الـjob يخلص من الـ2 |
