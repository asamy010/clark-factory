## 🐛 المشكلة المُبلَّغ عنها

**المستخدم**: "اخترت 5 فواتير وعملت ترحيل، اترحلوا وبعد 3 ثواني رجع 4 منهم تاني مسودة"

V19.54 كان فيه fix للـstale-closure في `upConfig`/`upSales`/`upTasks` (الـoptimistic state). الـUI كانت بتظهر الـ5 posted صحيح. لكن الـbug ظهر بشكل تاني: بعد 3 ثواني 4 منهم رجعوا draft.

## 🔍 الـRoot Cause الحقيقي

**Firestore writes بتشتغل parallel، last-to-land يكسب**.

```
سيناريو ترحيل 5 فواتير:
─────────────────────────
iter 1 → upConfigTx → setDoc(state1: invoice1=posted)   ──┐
iter 2 → upConfigTx → setDoc(state2: invoice1+2=posted) ──┤
iter 3 → upConfigTx → setDoc(state3: invoices1+2+3=p)   ──┤  parallel network
iter 4 → upConfigTx → setDoc(state4: invoices1-4=p)     ──┤
iter 5 → upConfigTx → setDoc(state5: ALL 5 posted)      ──┘

Network arrival order على Firestore (random):
  state5 lands first  → server: ALL 5 posted ✅
  state2 lands second → server: invoices 1+2 posted, 3+4+5 reverted to draft ❌
  state1 lands last   → server: ONLY invoice 1 posted ❌❌❌

Listener echos this back to the client → UI flips to draft for 4 invoices!
```

V19.54 fix على `configDocRef.current` كان بيخلي الـlocal state صحيح، لكن الـwrite-order على الـserver كان random. كل write بيكتب snapshot مختلف، وآخر write يصل = آخر state على الـserver.

## ✅ الإصلاح في V19.55

**Promise-chain serialization** للـwrites:

```js
const upConfigWriteQueueRef = useRef(Promise.resolve());

const upConfigTx = useCallback(async (...args) => {
  const _doWrite = async () => { /* ... existing setDoc logic ... */ };
  /* Chain onto the queue: write N starts only after write N-1 finishes */
  const myWrite = upConfigWriteQueueRef.current.then(_doWrite);
  upConfigWriteQueueRef.current = myWrite.catch(() => {}); /* keep chain alive on error */
  return myWrite;
}, [...]);
```

### النتيجة
- write 1 يبدأ
- write 2 يستنى لحد ما write 1 يخلص
- write 3 يستنى لحد ما write 2 يخلص
- ...
- Firestore receives writes IN ORDER → last call wins → state5 يفضل على الـserver

نفس الإصلاح طُبّق على:
- `upConfigTx` (factory/config)
- `upSalesTx` (factory/sales)
- `upTasksTx` (factory/tasks)

كل واحد له queue منفصلة — مفيش cross-doc blocking.

### Trade-off
- **Single-action operations**: غير محسوس (queue فاضي، الـwrite يبدأ فوراً)
- **Rapid sequential operations** (bulk-post، imports): الـwrites تشتغل sequential بدل parallel
- **الفايدة**: data consistency مضمونة 100% — مفيش lost writes تاني

## 🏷️ Feature: QC-2 (درجة تانية) لطباعة QR

في popup طباعة QR، اتضافت checkbox **⚠️ درجة تانية**.

### السلوك
لما الـcheckbox متعلم:
1. **QR يصغر ~80%** من حجمه الأصلي
2. **ختم "QC-2"** يظهر تحت الـQR في مستطيل صغير بحدود سوداء
3. **يطبّق على كل التابات الـ4**: يدوية / سيري / تلقائية / قطعة

### الـQR payload متغيرش
الـQR لسه فيه نفس البيانات (`CLARK:orderId:qty`) — الـscanner في المخزن بيشتغل عادي. الفرق فقط بصري للموظف اللي يقرأ الليبل.

### مثال الليبل
```
┌──────────────────────────┐
│  [LOGO أو CLARK]         │
│  3262115                 │
│  تويز اولادي قطعتين      │
│  مقاس: 6                 │
│                          │
│   ┌─ QR (شوية أصغر) ─┐  │
│   │                    │  │
│   │     [QR Code]      │  │
│   │                    │  │
│   └──────────────────┘  │
│      ┌─────────┐         │
│      │  QC-2  │         │  ← الجديد
│      └─────────┘         │
│                          │
│  سيري: 6                │
└──────────────────────────┘
```
