## Issue #1 (CRITICAL): Cash payment message ignored pending checks

### السياق

User report:

> "العميل ده سلم 3 شيكات والرصيد بيتحسب صح ومظبوط مع كل شيك بعد الخصم وتمام،
> بعدها سجلت دفعة 100 جنيه لنفس العميل. الرسالة وصلت للعميل ان الرصيد باقي
> محسبش الشيكات وده غلط طبعا. ركز في المشكلة دي وخد وقتك ضروري تتحل."

البيانات الفعلية (من الـ kashf):
- بيع: 1600 ج.م
- خصم 10%: -160 → after disc = 1440
- مدفوع نقدي: 200 ج.م (100 قديمة + 100 جديدة)
- مدفوع شيكات: 300 ج.م (3 شيكات × 100)
- إجمالي مدفوع: 500
- **الرصيد الفعلي: 940**

لكن رسالة الواتس آب بعد الدفعة الجديدة قالت `الرصيد المتبقي: 1,240` ج.م.

### السبب

الـ V19.76.2 fix طبّق الخصم بس ما طرحش الشيكات. الـ formula كانت:
```
balance = (gross − disc) − cashPayments
       = 1440 − 200 = 1240   ❌
```

كشف الحساب بيطرح الشيكات كمان:
```
totalPaid = cashPayments + receivableChecks   (في CustDeliverPg)
balance   = (gross − disc) − totalPaid
          = 1440 − 200 − 300 = 940   ✅
```

### الإصلاح في 5 أماكن

```js
// New unified formula — matches كشف الحساب
gross   = Σ(deliveries × price) − Σ(returns × price)
discAmt = round(gross × discPct/100)
checks  = Σ(receivable, non-bounced, non-cancelled, دفعة عميل)
balance = (gross − discAmt) − Σ(cashPayments) − checks
```

| Site | File | Line | Notes |
|---|---|---|---|
| 1 | `api/automation-tick.js` | `computeCustomerBalances` (~290) | Added 4th arg `checks` |
| 2 | `api/automation-tick.js` | `scanRecentPayments` (~450) | Loads checks + passes them |
| 3 | `api/automation-tick.js` | `scanRecentChecks` (~575) | Subtracts priorChecks before progressive batch |
| 4 | `src/pages/TreasuryPg.jsx` | Cash instant fire (~1080) | Subtracts `data.checks` |
| 5 | `src/pages/TreasuryPg.jsx` | Status-change computeBal (~2920) | Subtracts other checks; current handled by status |

For check scans (#3), the cron must NOT pass checks to `computeCustomerBalances`
or it would double-count the batch being processed. Instead it does an explicit
subtraction of checks where `batchId !== current.batchId` (or `id !== current.id`
for non-batch).

For status changes (#5), the formula subtracts ALL pending receivable checks
EXCEPT the one whose status is changing — that one's contribution depends on
the new status:
- `محصل` or re-presented `معلق` → counts as paid (subtract)
- `مرتد` → doesn't count (don't subtract — customer still owes)

## Issue #2: Treasury delete didn't cascade in legacy data

### السياق

> "الدفعة النقدي اللي متسجلة بـ100 جنيه انا حذفتها خالص ولسه موجودة في كشف الحساب.
> ياريت تأكد على عملية الحذف من الخزنة انها بتحذف جميع السجلات خالص."

`delTx` (TreasuryPg) used to cascade-delete `custPayments` only by:
- `treasuryTxId === tx.id` (forward link)
- `tx.custPaymentId === payment.id` (reverse link)

For payments saved before V15.9 (when `treasuryTxId` was introduced) OR after
a sync race that broke the link, neither match worked → the custPayment survived
the cascade and re-appeared in the kashf as an orphan.

### الإصلاح

Added legacy fallback — when the treasury tx is `category="دفعة عميل"` (or
`sourceType=cust_payment`) AND has `custId`, also remove orphan custPayments
matching `(custId, amount, date)`. Same for `supplierPayments`.

Risk of false positive: low — it requires an exact match on three fields. If
two payments happened on the same day with the same amount, only the unlinked
ones get removed (linked ones already filtered first).

```js
if(d.custPayments){
  // V19.20: link-based cascade (existing)
  d.custPayments = d.custPayments.filter(p =>
    p.treasuryTxId !== id && (!tx.custPaymentId || p.id !== tx.custPaymentId)
  );
  // V19.76.4: legacy-fallback (new)
  if (tx.custId && (tx.sourceType === "cust_payment" || tx.category === "دفعة عميل")) {
    d.custPayments = d.custPayments.filter(p => !(
      p.treasuryTxId === undefined &&
      p.custId === tx.custId &&
      Math.abs((Number(p.amount)||0) - (Number(tx.amount)||0)) < 0.01 &&
      p.date === tx.date
    ));
  }
}
```

## Issue #3: Local clock drift corrupts saved timestamps

### السياق

> "في مشكلة في التايم ستامب والتاريخ، انه بياخد تاريخ الكمبيوتر وده مش صح عشان
> في كمبيوتر بيكون مش مظبوط تاريخه او ساعته بيعمل مشاكل. عاوزه ياخد التاريخ
> والوقت من الانترنت بتاريخ ووقت القاهرة."

Every `new Date().toISOString()` call inherits the user's local clock. If the
computer is set 2 days back, all `createdAt` timestamps are 2 days back —
forever. The eventHistory chronology breaks, the kashf date columns lie, and
the WhatsApp `at:` timestamps in messages are wrong.

### الإصلاح

1. **New endpoint `api/now.js`**:
   ```
   GET /api/now → { now, nowMs, cairo, cairoDate, tz: "Africa/Cairo" }
   ```
   Vercel servers are NTP-synced; this is a trustworthy reference.

2. **New helper `src/utils/serverTime.js`** with:
   - `syncClock()` — fetches `/api/now`, computes RTT-aware skew (`skew = serverMs − (t0 + rtt/2)`)
   - `startClockSync()` — wired into App.jsx mount
   - `nowMs()` / `nowDate()` / `nowISO()` — return `Date.now() + skew`
   - `cairoDateStr()` — Cairo-tz YYYY-MM-DD via Intl
   - Re-syncs lazily after 30 min — long-lived sessions stay accurate
   - Falls back to plain `new Date()` on fetch failure (logged once)

3. **Replace `new Date()` in critical save paths**:
   - `TreasuryPg.jsx`: all `createdAt`, `updatedAt`, `at`, `editedAt`, `approvedAt`,
     and date initial values use `nowISO()` / `cairoDateStr()`
   - `CustDeliverPg.jsx`: same — including `addCustPayment` which is the path
     the user's report came from

Display-only formatters (e.g. `toLocaleString("ar-EG")` for print headers) are
left as plain `new Date()` since they're transient UI strings, not persisted.

### Caveats

- The first render after page load uses `skew=0` (sync hasn't completed).
  Most save actions happen seconds-to-minutes after load, by which time sync
  has completed. Fast-clicking-save-immediately-after-page-load is a rare edge
  case where local time leaks in.
- We don't force a re-render when sync completes, to avoid a global re-render
  on every page load. The `today` constant inside renders gets re-evaluated
  on the next state change.
