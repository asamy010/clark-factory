## التغييرات

### 1. Backfill prevention via `enabledAt`

**Behavior change:**
- كل event عنده دلوقتي `enabledAt` field (ISO timestamp)
- يتعمل set في كل مرة الـuser يـtoggle ON
- الـscan functions بـskip أي entity created BEFORE الـenabledAt

**Example:**
- 14:00 — user enables `paymentReceived` trigger
- enabledAt = "2026-05-05T14:00:00Z"
- 14:05 — cron tick scans
- Sees a payment created at 13:30 (yesterday's payment, just within 24h window)
- 13:30 < 14:00 → SKIP
- Sees a payment created at 14:03 (after enable)
- 14:03 >= 14:00 → FIRE

**Implementation:**
- `src/pages/AutomationPg.jsx`: `toggleEvent` sets `enabledAt = new Date().toISOString()`
- `api/automation-tick.js`: `entityTs(entity)` extracts createdAt > recordedAt > date
- All 4 scan functions check `entityTs(entity) >= enabledTs` else skip

**Backward compat:**
- Users who enabled triggers in V19.70/V19.70.1 don't have `enabledAt`
- `ensureEnabledAt(db, eventType, currentValue)` auto-sets it on first scan
- That first scan is skipped entirely (no burst of historical events)
- Subsequent scans use the timestamp normally

### 2. Customer balance computation

**Before:**
```js
balance: Number(p.balanceAfter) || 0
// p.balanceAfter is never set during payment recording → always 0
```

**After:**
```js
const balances = computeCustomerBalances(orders, payments);
// ...
balance: Math.round(balances[p.custId] || 0)
```

**Formula** (matches `_alertsSection` in `buildDailyReport.js`):
```
balance = Σ(deliveries.qty × price) − Σ(returns.qty × price) − Σ(payments.amount)
```

Sign convention: positive = customer still owes; negative = credit on account.

**Why "current balance" not "balance immediately after this specific payment":**
- Computing exact running balance per payment requires sorted-iteration through all events
- Most users want to know "after this payment, how much do I still owe?" which is current balance (since this payment is the latest event for the customer typically)
- The 24h window means we're only firing for very recent payments anyway

### 3. Performance optimization

**Before:** Each of the 3 order-dependent scans (sale/payment/lateOrder) loaded orders separately = 3× Firestore reads per tick.

**After:** Main handler loads orders ONCE if any order-dependent scan is enabled, passes the cached array to scan functions. 1× Firestore read per tick when 1+ such triggers are enabled.

```js
let ordersCache = null;
const needsOrders = saleEnabled || paymentEnabled || lateEnabled;
if (needsOrders) ordersCache = await loadActiveOrders(db, cfg);
// ...passes to each scan as 5th arg
```

## Edge cases handled

| Scenario | Behavior |
|---|---|
| Trigger never enabled | No scan, no fire. |
| Trigger enabled, no entities yet | Scan runs, scans 0 entities. |
| Trigger enabled at 14:00, entity from 13:30 | Skipped (skippedOld counter increments). |
| Trigger enabled at 14:00, entity from 14:00:01 | Fired (createdAt >= enabledAt). |
| User toggles OFF then ON again at 16:00 | enabledAt updated to 16:00. Old entity from 14:30 now skipped. |
| Existing user upgrades from V19.70.1 (no enabledAt) | First scan auto-sets enabledAt = now, skips that scan. Next scan works normally. |
| Entity has no `createdAt`/`recordedAt`/`date` | Skipped (treated as unknown-age, conservative). |
| Backdated entity (date=yesterday but createdAt=today) | createdAt is checked first → fired correctly (it WAS recorded today). |

## Important user note

**For existing triggers (enabled in V19.70 / V19.70.1):**

The first cron tick after upgrading will:
1. See no `enabledAt` on the event config
2. Auto-set `enabledAt = now`
3. Skip that tick (return `reason: "enabledAt-just-set"`)

Subsequent ticks scan normally with the new timestamp as the cutoff. **No historical events will be fired.**

If the user explicitly wants to backfill events from before today, they can manually edit Firestore: set `automation.eventTriggers.events.{eventType}.enabledAt` to a past timestamp.
