## السياق

V19.69.5 وصلت الـVPS cron LIVE — الـscheduled daily report بيشتغل تلقائياً في الميعاد المحدد. V19.70 بتضيف الطبقة الجاية: **event-driven** triggers — الـsystem يبعت WhatsApp فوراً لما حدث مهم يحصل (بيع، دفعة، إلخ) بدون انتظار schedule.

## ⚙️ الـ4 events المدعومة

| Event | Trigger | Recipients | Detection |
|---|---|---|---|
| 💰 **saleCompleted** | عند تسجيل customerDelivery | customer + owner | Cron scan آخر 24 ساعة (5-min granularity) |
| 💵 **paymentReceived** | عند تسجيل custPayment | customer + owner | Cron scan آخر 24 ساعة |
| ⚠️ **lateOrder** | أوردر بدون activity ≥ N أيام | owner (+ customer optional) | Cron daily scan، 1 alert per order per day |
| 📅 **checkDue** | شيك مستحق ضمن N أيام | owner | Cron daily scan، 1 alert per check per day |

كل event عنده:
- Toggle ON/OFF (default: OFF)
- Recipients selectable per role (customer/owner)
- Template editable per role مع variables
- Threshold (للـcron-only events)
- Test/preview قبل الـenable

## 🟢🟡 الـModes: Auto vs Manual

في الـAutomation page → Tab "🔥 Triggers الفورية" → mode toggle فوق:

### 🟢 Auto mode (الـdefault)
الـsystem يبعت تلقائياً:
- الـcron tick كل 5 دقائق يـscan الـ4 paths
- لو لقى event جديد → يبعت عبر الـbridge
- لو الـbridge فشل → الـentry يـqueue في pending → الـcron يـretry next tick
- بعد 5 محاولات فاشلة → الـentry يـmark كـ"need manual review" بـred

### 🟡 Manual mode
الـsystem ميـبعتش تلقائياً:
- كل event جديد يـqueue في pending list
- الـuser يفتح الـTriggers tab → يشوف الـpending queue
- يضغط "إرسال" على كل rsali واحدة (force-fire)، أو "إرسال الكل" للـbatch
- مفيد لو فيه مشكلة في الـserver وعايز تتحكم باليد

**النصيحة:** سيب على Auto. خد Manual بس لو لاحظت messages بتـفشل وعايز تحقق قبل ما تتبعت.

## 🛡️ Idempotency — لا توجد رسائل مكررة

كل event عنده `idempotencyKey` unique:
- saleCompleted: `sale:${orderId}:${date}:${qty}:${custId}`
- paymentReceived: `payment:${paymentId}`
- lateOrder: `lateOrder:${orderId}:${cairoDate}`
- checkDue: `checkDue:${checkId}:${cairoDate}`

الـ`eventHistory[]` (cap 100) بيـtrack كل الـfired events. قبل أي send: لو الـkey موجود = skip (return `deduped: true`).

النتيجة: حتى لو الـclient وصلها واتـcron الاتنين شافوا نفس الـevent، الـsecond محاولة تـskip تلقائياً.

**Override:** الـ"Send Now" button في الـpending queue يـpass `force: true` — يتجاوز الـidempotency. مفيد لو عايز تـresend رسالة قديمة باليد.

## 🏗️ Architecture

### Files

```
api/
├── event-trigger.js           # POST endpoint (thin HTTP wrapper)
├── _eventProcessor.js         # Core: processEvent() — validate + idempotency + bridge + history
├── _eventBuilder.js           # Pure: substituteTemplate + buildEventMessages + samplePayload
└── automation-tick.js         # Cron tick: pending drain + 4 scan helpers

src/
├── pages/AutomationPg.jsx     # NEW Triggers tab + sub-components (TriggersTab, EventCard, etc.)
└── utils/automation/
    ├── eventBuilder.js        # Mirror of api/_eventBuilder.js (client-side use)
    └── buildDailyReport.js    # Extended DEFAULT_AUTOMATION_CONFIG with eventTriggers
```

### Flow: client records sale → instant WhatsApp

```
DetPg.jsx          API                   Firestore           Bridge          Phone
─────────────      ─────                 ─────────           ──────          ─────
record sale  ──→   client write
                                          deliveries[].push
                                              ↓
                                          (5 min)
                                              ↓
              ←── automation-tick (cron) ── scan
              processEvent("saleCompleted", payload, key)
                       ↓
                   isAlreadyFired? no
                   eventCfg.enabled? yes
                   resolvePhones (owner + customer.phone)
                   buildMessages (substitute templates)
                       ↓
                   bridgeSend ─────────────────→ /send  ───────→ WhatsApp
                       ↓
                   recordResult (eventHistory)
                       ↓
                   pending.delete (if any)
```

### Flow: bridge fails → auto retry

```
processEvent
    ↓
bridgeSend → throws (network error)
    ↓
queuePending (with attempts:1, lastError)
    ↓
recordResult (success: false)
    ↓
return { ok: false, queued: true }

(5 min later)

automation-tick
    ↓
loop pending.filter(p => p.attempts < 5)
    ↓
processEvent (retry)
    ↓
on success → pending.delete + history.push
on failure → pending.attempts++ (try again next tick)
```

## 📊 Performance notes

### Per-tick cost
- pendingDrain: max 10 entries (capped to avoid timeout)
- saleCompleted scan: max 50 deliveries (24h window)
- paymentReceived scan: max 50 payments (24h window)
- lateOrder scan: O(orders × deliveries) but only fires after threshold
- checkDue scan: O(checks) but only fires within thresholdDays window

### Firestore reads per tick
- factory/config: 1 read (always)
- orders: 1 collection read (for late + sale scans, when enabled)
- checks: 1 collection read (for checkDue, when enabled)
- customers: 1 partitioned read (when customer recipient enabled)

**Estimate:** ~5 reads per tick. With 5-min ticks = 1,440 reads/day. Within Firestore free tier (50K reads/day).

### Bridge calls per tick
- 1 per fired event (rarely batched). Anti-ban delays handled by the bridge itself.

## 🐛 Troubleshooting

### "الـevent ما اتـfireش"
1. تأكد الـevent enabled في الـUI
2. تأكد الـrecipients مختارين (لو الـcustomer مش tickيد، الـcustomer ميستلمش)
3. تأكد الـowner phones موجودة لو الـowner recipient on
4. شوف الـEvent history في الـAutomation → History tab — لو الـevent fired بـsuccess: false، الـerror هيبقى موجود
5. شوف الـpending queue — لو الـevent stuck فيها، الـlastError هيقولك الـcause

### "Bridge unreachable"
- الـpending queue بتـpopulate تلقائياً
- لما الـbridge يرجع، الـcron next tick هيـdrain
- لو عايل تتأكد، الـ"إرسال الكل" زرار في الـpending queue يـforce-fire الكل

### "Idempotency blocking re-send"
- استخدم الـ"Send Now" زرار على الـpending entry (يبعت بـforce: true)
- أو امسح الـentry من الـeventHistory في Firestore Console يدوياً (rare case)

### "I want to disable a specific event"
- اقفل الـtoggle في الـcard. الـcron هيـskip تلقائياً (`reason: "event-disabled"`).

### "I want to test a template before enabling"
- افتح الـcard → اختار الـrecipient → اضغط 👁 معاينة → الـsample-substituted text هيظهر في الـgreen panel

## 🚧 Known limitations / V19.71 candidates

1. **Portal link generation in cron path** — الـcron-detected sale/payment events حالياً ما بيـgenerate portal links (الـ`{portalLink}` placeholder بيفضل فاضي). الـclient path يقدر يـpre-generate. **Future:** يـcall `/api/customer-portal-sign` من cron مباشرة.
2. **No client-side instant-fire hooks** — الـcron-only approach يضيف 0-5 min latency. **Future:** add hooks في DetPg/TreasuryPg للـzero-latency fire، الـcron يبقى fallback.
3. **No per-customer opt-out** — مفيش way الـcustomer يقول "ما تبعتليش messages تلقائية". **Future:** integrate مع `data.campaignBlocklist[]` (V19.29).
4. **eventHistory cap 100** — لو volume عالي جداً (>100 events/day)، الـold idempotency keys بتروح. الـ24-hour scan filter بيـmitigate ده. **Future:** dedicated Firestore collection للـeventNotifications.
5. **No analytics on event delivery** — مفيش رؤية لـ"كم بيع تم نوتيف، كم وصل، كم فشل". **Future:** dashboard للـper-event metrics.
