# V19.70.20 — Available stock breakdown popup

**Date:** 2026-05-06
**Type:** feature
**Status:** shipped

## What

The `رصيد متاح` card in the CustDeliverPg dashboard is now clickable. Clicking it opens a popup that:

- Lists every model with `avail > 0`
- For each model, breaks down available pieces into **سيري** (full sets, depleted first by sales) and **كسر** (loose pieces)
- Shows the rack-size factor when relevant (e.g. `24` series pieces displayed as `24` plus a subtitle `4×6` meaning 4 series sets of 6 each)
- Supports search/filter by model number or description
- Has two action buttons:
  - **🖨 طباعة** — browser-native print
  - **📤 إرسال PDF واتساب** — sends a PDF + text summary to every phone in `data.automation.eventTriggers.ownerPhones`

## Why

The dashboard tile previously showed only a total (e.g. `666`) with no way to drill into which models contribute to that number. To make a stock decision (which models to push, which to replenish, which to mark as stale), the user had to open the matrix popup and scroll. The popup makes the breakdown one click away with a print/share path included.

## Series-vs-broken split logic

`stockModels[i]` already has `seriesQty`, `brokenQty`, and `custDel` (net sold). Sales deplete series stock first (matching the existing computation in the matrix's رصيد متاح للبيع row). So:

```js
availSeries = max(0, seriesQty - custDel)
availBroken = avail - availSeries  // = brokenQty when series isn't fully consumed
```

When `rackSize > 0`, the series count also displays as `floor(availSeries / rackSize) × rackSize` (e.g. `4×6 = 24`).

## PDF generation

Uses `htmlToPdfBase64` from V19.70.12 (lazy-loaded html2canvas + jsPDF). The HTML structure uses **`<td class='h'>` for table headers** instead of `<th>` — the V19.70.19 fix that bypasses html2canvas's Arabic-shaping bug for header cells. The `.h` class carries a gradient + bold styling identical to the old `<th>` styling, plus the orange/amber theme to differentiate the report from delivery receipts.

`buildReportHTML` is a single function used by both the print path and the PDF path — guarantees no drift between what the user prints and what gets sent over WhatsApp.

## WhatsApp send flow

1. Pull `ownerPhones` from `data.automation.eventTriggers.ownerPhones`. If empty, abort with toast.
2. Pull `bridgeUrl` + `bridgeToken` from `data.campaignBridge`. If `bridgeUrl` empty, abort.
3. Confirmation prompt showing the count of recipients.
4. Lock the popup (`sending: true`).
5. Build PDF base64 once.
6. For each phone, sequential await `bridge /send` with:
   - `phone`
   - `message` (text summary: totals + top 5 models)
   - `media: [{base64, mime: "application/pdf", name}]`
7. Toast the success/fail counts when done.

The sequential await preserves the bridge's anti-ban delays (8-15 sec between sends).
