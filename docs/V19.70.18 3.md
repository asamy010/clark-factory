# V19.70.18 — Drawer name + customer due reminder

**Date:** 2026-05-06
**Type:** feature
**Status:** shipped

## Two related features in one version

The user requested two changes that work together:

1. A new field on receivable checks: **اسم صاحب الشيك** (drawer name — the name printed on the check). Useful when a customer pays us with a 3rd-party check, and we need to know whose bank account will be debited at presentation.
2. An **automatic WhatsApp reminder** sent to the customer before the check's due date, so they can ensure their bank account has funds. Threshold (number of days) is user-configurable.

## Feature 1: drawerName field

### UI

A new input appears in the check form, **only when type is "receivable" (أوراق قبض)**. For payable checks (أوراق دفع), we are the drawer — the field is meaningless and stays hidden.

The placeholder hints at the default fallback: if the user leaves it blank, we treat it as the party name (`ch.party`).

### Storage

- New optional field `drawerName` on the check object.
- Set to `null` for payable checks.
- Defaults to `ch.party.trim()` for receivable when the user leaves it blank.

### Surfaced in:

- **Endorse popup** — the supplier sees a `✍️ صاحب الشيك (المكتوب عليه): X` line when X differs from the party we received the check from. This avoids confusion when the supplier tries to track who owes the bank.
- **Event payloads** — every check-related event (`checkDue`, `checkPaymentReceived`, `checkPaymentIssued`, `checkBounced`, `checkCollected`, `checkEndorsed`, `checkRePresented`) now includes `{drawerName}` as an available template variable.

### Backward compatibility

Legacy checks (created before V19.70.18) don't have `drawerName`. The cron's payload builder falls back to `partyName` so existing data still produces correct messages. No migration needed.

## Feature 2: customer due-reminder

### Mechanism

The existing `checkDue` cron already scans all pending receivable + payable checks and fires events when the due date is within `thresholdDays`. V19.70.18 extends it:

- `EVENT_VARIABLES.checkDue.recipientRoles` now includes `"customer"` alongside `"owner"`.
- `DEFAULT_AUTOMATION_CONFIG.events.checkDue.recipients.customer` defaults to `false` (opt-in — existing users don't suddenly start blasting their customers).
- A new `customer` template is included in the defaults — bilingual-friendly Arabic, references `{drawerName}` so the customer can identify which specific check we mean.
- The cron logic checks `c.type === "receivable"` before firing the customer-side message (payable doesn't have a meaningful customer recipient).

### Idempotency keys per recipient role

The pre-existing `checkDue` key was `checkDue:${id}:${date}` — fires once per check per day for the owner. Adding the customer recipient with the same key would either:
- Overwrite the owner fire's success → re-fire owner next tick, OR
- Get deduped by the owner's eventHistory entry → never fire customer.

**Fix:** the cron now fires owner + customer as **two separate `processEvent` calls** with distinct keys:
- Owner: `checkDue:${id}:${date}` (legacy key, preserved for back-compat with existing eventHistory)
- Customer: `checkDue:${id}:${date}:customer`

A new `recipientFilter: ["owner"]` (or `["customer"]`) parameter is passed to `processEvent` to limit which messages `buildEventMessages` produces for that call. Legacy callers (without `recipientFilter`) keep their original behavior.

### User configuration

The trigger card in AutomationPg → Triggers tab now shows a `👤 العميل` toggle for `checkDue` automatically (the UI dynamically reads `EVENT_VARIABLES.checkDue.recipientRoles`). The same `thresholdDays` input governs both owner and customer alerts.

### Customer template (default)

```
🔔 *تذكير: شيك يستحق الصرف قريباً*

مرحباً {customerName}،
نود تذكيركم بأن الشيك التالي مستحق الصرف من البنك خلال *{daysToDue}* يوم:

✍️ صاحب الشيك: {drawerName}
🏦 البنك: {bank}
#️⃣ رقم الشيك: {checkNo}
💰 القيمة: {amount} ج.م
📆 تاريخ الاستحقاق: {dueDate}

⚠️ يرجى التأكد من تغطية الشيك في الحساب البنكي قبل تاريخ الاستحقاق لتجنب ارتداد الشيك.

شكراً لتعاملكم 🌟
```

User-editable from the Triggers UI. The default focuses on the practical action ("cover the check at the bank") rather than just informational details — matches the user's request rationale.

## Test plan

### drawerName field
1. Open Treasury → الشيكات tab → create a receivable check.
2. Verify the new `اسم صاحب الشيك` input appears.
3. Leave it blank → save. Open the check from edit. Should show `drawerName = party name` (the fallback).
4. Edit and set a different name. Save. Re-open. Drawer name should match what you entered.
5. Switch to "أوراق دفع" — input should disappear.

### Endorse popup
1. Create a receivable check with `drawerName` set to something different from the party.
2. Open the endorse popup → verify the `✍️ صاحب الشيك (المكتوب عليه)` line appears.
3. If `drawerName === party`, the line should be hidden (no info value).

### Customer due-reminder
1. Open Automation → Triggers tab → expand `📅 شيك يستحق قريباً`.
2. Verify `👤 العميل` recipient toggle appears.
3. Toggle it on. Set `thresholdDays = 1` for fast testing.
4. Create a receivable check with dueDate = tomorrow, attach to a customer with a phone.
5. Wait for the next cron tick (or trigger automation-tick manually). Customer should receive the reminder. Owner gets the existing reminder.
6. Re-fire the same tick — the eventHistory should dedupe both. No duplicate sends.

### Backward compat for old templates
1. If a user customized their `checkDue.owner` template before V19.70.18, the new `{drawerName}` variable is missing from their template. That's expected — they can add it manually if they want.
2. Old checks without a `drawerName` field still work — the cron falls back to `partyName`.
