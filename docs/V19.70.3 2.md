## التغيير

### Client-side instant hook في TreasuryPg

```
User saves treasury tx (دفعة عميل)
  ↓
upConfig() commits the data
  ↓
[NEW V19.70.3] fire-and-forget POST to /api/event-trigger
  ↓ (in background, doesn't block save UX)
Endpoint validates + builds messages + sends via bridge
  ↓
Customer's WhatsApp lights up — within seconds, not minutes
```

### الـCron يفضل fallback

```
Cron tick (every 5 min)
  ↓
Scan recent payments (last 24h)
  ↓
For each: check eventHistory for `payment:${id}` key
  ↓ found (because client already fired) → SKIP
  ↓ NOT found (client failed)            → fire now
```

الـidempotency `payment:${id}` يضمن:
- ✅ Client succeeded → cron skips on next tick
- ✅ Client failed silently → cron picks up within ≤5 min
- ✅ Both fire (race) → second one returns `deduped: true`

## Implementation details

### Pre-generated paymentId

قبل V19.70.3 الـ`gid()` كان inline في الـpush. لـclient-side fire، محتاج أعرف الـid الجديد للـidempotencyKey. الحل:

```js
const _instantPay_needed = (linkedCustId && txType==="in" && txCategory==="دفعة عميل" && !editId);
const _instantPay_id = _instantPay_needed ? gid() : null;
const _instantPay_customer = _instantPay_needed ? customers.find(x=>x.id===linkedCustId) : null;

upConfig(d => {
  // ...
  d.custPayments.push({id: _instantPay_id || gid(), /* ... */});
  // ...
});
```

### Fire conditions (all 4 must be true)

| Condition | Why |
|---|---|
| `txType === "in"` | Only incoming payments |
| `txCategory === "دفعة عميل"` | Only customer payments (not supplier/salary/etc) |
| `linkedCustId` truthy | Customer must be linked (not orphan tx) |
| `!editId` | Only NEW entries (not edits) |
| `customer.phone` truthy | Customer must have a phone to message |

أي واحدة missing → skip silently. الـcron مش هيـfire برضه في الـcase ده — الـtrigger config هي اللي بتقرر، مش الـclient.

### Balance computation client-side

نفس الـformula اللي الـcron يستخدمها — `deliveries × price − returns − payments`. الـclient بيشتغل من `data` prop (latest state), بيـsubtract الـnew amount manually:

```js
let bal = 0;
for (const o of data.orders || []) {
  for (const d of (o.customerDeliveries || [])) {
    if (d.custId === linkedCustId) bal += qty × price;
  }
  // ... returns ...
}
for (const p of data.custPayments || []) {
  if (p.custId === linkedCustId) bal -= p.amount;
}
bal -= amt;  // the new payment we just recorded
```

### Fire-and-forget pattern

```js
(async () => {
  try {
    const idToken = await user.getIdToken();
    await fetch("/api/event-trigger", { /* ... */ });
  } catch (e) {
    // Silent — cron will retry
    console.warn("[V19.70.3] instant fire failed (cron will retry):", e.message);
  }
})();
```

- لا يحبس الـsaveTx (الـuser يشوف الـsave حصل فوراً)
- لو الـnetwork اتقطعت: silent error، الـcron next tick يـcatch up
- لو الـapp اتقفلت بعد ما الـrequest startt: الـbrowser يكمل الـrequest عادة (لكن لو فيه issue، الـcron يـbackup)

## What this does NOT do

- ❌ **Sale-completed instant fire**: still cron-only. The user's stated use case is collector recording payments — sale is less time-sensitive (DetPg/CustDeliverPg flows). Future V19.70.4 if requested.
- ❌ **Edit-time fire**: editing an existing payment doesn't re-fire. The original event was already notified; editing the amount/method shouldn't generate a new "you paid X ج.م" message. If the user explicitly wants notification on edit, that's a separate feature.
- ❌ **Late order / Check due instant**: these are state-based daily scans by definition — no client-side trigger makes sense.
