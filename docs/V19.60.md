## 🔍 الـRoot Cause الحقيقي

`rebuild()` في الـpartitioned listener كان بيكتب **كل الـfields** في كل listener fire:

```js
const next = {};
for (const f of PARTITIONED_FIELDS) {
  next[f] = flatten(docsById[f], pendingMap);  // empty Map → []
}
setPartitionedData(next);   // ← يـoverwrite الـcache!
```

**سيناريو الـbug**:
1. App mount → partitionedData hydrated من localStorage (35 عميل) ✅
2. workshops listener يـfire الأول → `rebuild()`:
   - `next.customers = flatten(empty Map) = []` ← لأن customers listener لسه ما اشتغلش
   - `setPartitionedData({customers: [], workshops: [...], ...})` ← **مسح الـcache!**
3. partitionedDataRef.current.customers = [] (via useEffect sync)
4. User يعمل بيع سريع → upConfig:
   - `explicitPartBefore.customers = ref.customers = []`
   - بعد fn(next): newPart.customers = [] (السلة ما لمستش customers)
   - `setPartitionedData(newPart)` ← يثبّت الفراغ في الـstate
5. **القائمة تختفي**

كل بيع = نفس السيناريو لأن الـref بقى عنده empty customers.

---

## ✅ الـFix: Field-isolated rebuild

`rebuild()` بقى يحدّث **فقط الـfields اللي listener بتاعها اشتغل (`firstFires[f] === true`)**:

```js
setPartitionedData(prev => {
  const next = { ...prev };
  for (const f of PARTITIONED_FIELDS) {
    if (firstFires[f]) {
      next[f] = flatten(docsById[f], pendingMap);
    }
    /* else: keep prev[f] (cache or last-good value) */
  }
  return next;
});
```

### النتيجة
- workshops listener يـfire → workshops يتحدّث، customers يفضلوا 35 من الـcache ✅
- لما customers listener فعلاً يـfire → `firstFires.customers = true` → يتحدّث بالـfresh data
- البيع يستخدم ref عنده 35 customers → newPart.customers = 35 → state يفضل 35
- **القائمة ما تختفيش أبداً** ✅

### طُبّق الإصلاح على 4 rebuild functions
1. **partitioned** rebuild (master data — هو اللي ظهر فيه الـbug)
2. **split** rebuild (treasury, custPayments, إلخ)
3. **sales-split** rebuild (packages, custDeliverySessions)
4. **tasks-split** rebuild (tasks, stickyNotes, inventoryAudits)

نفس الـpattern في كلهم — أصح architecturally.

## 🎯 التوضيح التقني

### Why V19.59 alone wasn't enough
- V19.59 شيل الـ`partitionedLoaded` gate من الـuseMemo
- لكن الـstate نفسها كانت بتـoverwrite بفراغ بسبب rebuild
- الـmerge كان بيستخدم state فاضية → empty merged data
- V19.60 يصلح الـsource: state تفضل عامرة بـcache لحد ما الـactual data تجي

### Field isolation principle
- كل listener owns حقل واحد فقط
- مالوش حق يكتب على fields أخرى
- `firstFires[f]` يضمن إن الـfield ميتحدّثش إلا لما الـowner listener بتاعه يفير
- ده correctness layer essential — مش بس optimization
