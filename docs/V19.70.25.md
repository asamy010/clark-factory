# V19.70.25 — Bundled Tajawal TTF (no CDN dependency)

**Date:** 2026-05-06
**Type:** fix (critical)
**Status:** shipped — should resolve the PDF lib load failures

## The CDN war we lost

Three versions tried to fetch Cairo TTF from CDNs. All failed:

| Version | URL | Outcome |
|---------|-----|---------|
| V19.70.23 | `cdn.jsdelivr.net/gh/google/fonts@main/.../Cairo-Regular.ttf` | **403** — Google Fonts repo blocks raw passthrough |
| V19.70.24 | `cdn.jsdelivr.net/npm/@fontsource/cairo@5.0.13/files/cairo-arabic-400-normal.ttf` | **404** — fontsource v5 ships only WOFF/WOFF2, no TTF |
| V19.70.24 fallback | `unpkg.com/@fontsource/cairo@5.0.13/files/cairo-arabic-400-normal.ttf` | **CORS-blocked** — unpkg has no CORS headers |
| V19.70.24 fallback | `cdn.jsdelivr.net/npm/@fontsource/cairo@4.5.13/...` | **404** — same as v5 |

The user's PDF generation was completely broken.

## The right solution: bundle the TTFs

After confirming via `api.github.com/repos/google/fonts/contents/ofl/cairo` that Cairo on Google Fonts is now ONLY available as a variable font (`Cairo[slnt,wght].ttf`, ~600KB single file), I switched to **Tajawal** — another modern Arabic sans-serif on Google Fonts that DOES still ship as static TTF files for individual weights.

The TTFs (Regular + Bold, ~60KB each, total ~120KB) are now bundled in `public/fonts/`. Vite copies the `public/` directory to `dist/` at build time — same origin, served from the app itself, zero CDN dependency, zero CORS issues, zero 404s.

```
public/fonts/Tajawal-Regular.ttf  60 KB
public/fonts/Tajawal-Bold.ttf     60 KB
```

The `arabicPdf.js` utility now fetches:

```js
const TAJAWAL_REGULAR_URL = "/fonts/Tajawal-Regular.ttf";
const TAJAWAL_BOLD_URL    = "/fonts/Tajawal-Bold.ttf";
```

These are RELATIVE URLs that resolve to the app's own origin. On `clark-factory.vercel.app`, they hit `clark-factory.vercel.app/fonts/Tajawal-Regular.ttf` — zero external requests.

## Why Tajawal vs Cairo

- **Both are modern Arabic sans-serif** — visually similar, designed for screen + print
- **Cairo on Google Fonts** = variable font only (~600KB single file, jsPDF v2.5.1 has limited variable font support)
- **Tajawal on Google Fonts** = static instances (Regular, Bold, Black, etc. as separate TTFs)

For our use case (rendering a delivery receipt with Arabic text in two weights), Tajawal's static TTFs are the practical choice.

## Backward compatibility

The font family is still aliased as `"Cairo"` in jsPDF's font registry:

```js
pdf.addFileToVFS("Tajawal-Regular.ttf", base64);
pdf.addFont("Tajawal-Regular.ttf", "Cairo", "normal");
pdf.addFileToVFS("Tajawal-Bold.ttf", base64);
pdf.addFont("Tajawal-Bold.ttf", "Cairo", "bold");
pdf.setFont("Cairo");
```

So all the `pdf.setFont("Cairo")` calls in `buildAvailableStockPdfBase64` and `buildDeliveryReceiptPdfBase64` continue to work without modification. The visual font is Tajawal but the API contract is unchanged.

## Performance

- **CDN approach (V19.70.23/24):** 3-5 seconds for first PDF generation (downloading ~400KB across 2 CDN requests, possibly cold cache)
- **Bundled approach (V19.70.25):** ~500ms-1s for first PDF (~120KB from Vercel's edge cache, hot path)

Subsequent PDFs in the same session: instant (font data cached in JS module state).

## Trade-off: app deploy size

Adds ~120KB to `dist/` deployment. Negligible — Vercel handles this transparently.

## What didn't work + why

- Variable Cairo TTF (~600KB): single file with weight axis. jsPDF v2.5.1 doesn't reliably extract individual weights from a variable font — it just renders at the file's default weight. Both "normal" and "bold" calls would render the same weight. Tajawal's two static files give us proper visual contrast.

- Inline base64-encoded TTF in JS: would have added ~160KB (post-base64) to the main bundle. Tajawal as separate files is cleaner — Vite serves them directly without embedding in JS.

## What remains as future work

- Test with real Arabic content. The shaper from V19.70.21 should handle common Arabic letter forms; edge cases (specific Persian/Urdu characters, ligatures beyond Lam-Alef) may need adjustments to `_AR_FORMS` in `arabicPdf.js`.
- If a user complains about the Tajawal look (vs Cairo specifically), we can add Cairo's variable font as an opt-in — at the cost of ~5x larger TTF files.
