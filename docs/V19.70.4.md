## Phase A: Sale instant fire

### Pattern (same as V19.70.3 paymentReceived)

```js
// Pre-generate IDs before upConfig
const deliveryIds = items.map(() => gid());

// Use them in the push
o.customerDeliveries.push({ id: deliveryIds[idx], /* ... */ });

// After save success: fire-and-forget for each item
(async () => {
  const idToken = await user.getIdToken();
  await Promise.all(items.map((it, idx) => 
    fetch("/api/event-trigger", {
      method: "POST",
      headers: { /* ... */ },
      body: JSON.stringify({
        eventType: "saleCompleted",
        payload: { customerName, qty, modelNo, value, date, salesperson, portalLink: "" },
        customerPhone: customer.phone,
        idempotencyKey: "sale:" + deliveryIds[idx],
      }),
    }).catch(() => {/* silent — cron fallback */})
  ));
})();
```

### Idempotency consistency

**Before V19.70.4** the cron used composite keys:
```js
`sale:${o.id}:${date}:${qty}:${d.custId || "x"}`
```
Risk: collision if same customer buys same qty same day same model.

**After V19.70.4**: prefer `entry.id` if available:
```js
d.id ? `sale:${d.id}` : `sale:${o.id}:${date}:${qty}:${d.custId || "x"}`
```
The client always pre-generates id, so client + cron always match.

## Phase B: Campaign scheduling

### Data model

```js
data.scheduledCampaigns = [
  {
    id: "sched_xxx",
    templateId, templateName, templateBody,
    segmentKey, segmentLabel,
    items: [{id, name, phone}],/* audience snapshot */
    scheduledAt: "2026-05-06T10:00:00.000Z",/* ISO */
    status: "scheduled" | "firing" | "done" | "failed" | "cancelled",
    createdAt, createdBy,
    sendMode: "bridge",
    images: [],/* placeholder for V19.70.5 */
    sentCount: 0,
    failedCount: 0,
    error: null,/* set if status === "failed" */
    completedAt: null,/* set when done/failed */
    firingStartedAt: null,/* set when claimed by a cron tick */
  }
]
```

### Flow

```
User: New Campaign → pick template → pick segment
  ↓
[ChooseSendMode screen]
  ↓ click "📅 جدولة لوقت لاحق"
  ↓
[Inline datetime picker + Save]
  ↓ defaults to "now + 1 hour"
  ↓ on Save:
upConfig: data.scheduledCampaigns.unshift(newEntry)
  ↓
Redirected to "📅 المجدولة" tab
  ↓
... time passes ...
  ↓
VPS cron tick (every 5 min)
  ↓
scanScheduledCampaigns:
  - find oldest "scheduled" with scheduledAt <= now
  - transactional claim (status="firing") — prevents double-fire
  - build messages from templateBody + items
  - POST bridgeUrl/send
  - mark "done" with sentCount, OR "failed" with error
  ↓
UI shows updated status (live via Firestore listener)
```

### Cancel flow

User clicks "❌ إلغاء" on a scheduled entry:
- Status → "cancelled"
- Cron skips it (only fires "scheduled")

### UI: ScheduledCampaignsList

New component appended to CampaignsPg.jsx. Shows:

- Table: قالب · جمهور · ميعاد · حالة · عدد · إجراء

Status pills color-coded:

- 🟠 scheduled (overdue) — متأخر / قيد التنفيذ
- 🟡 firing — جاري الإرسال

## What's NOT in V19.70.4 (deferred)

### Image attachments

User requested: "حملة برسالة وصور"

The text part works in V19.70.4. For images, the data model has `images: []` placeholder, but actual image support needs:

1. **Client-side**: image picker, upload to Firebase Storage, get URL/path
2. **Cron-side**: fetch image from Storage, base64-encode, attach to bridge payload
3. **Bridge-side** (clark-wa-bridge/server.js): accept `mediaBase64 + mediaMime + mediaName` per message, build `MessageMedia` object, send via whatsapp-web.js

**The bridge change requires SSH access + Docker rebuild on the Contabo VPS.** Out of scope for V19.70.4 (Vercel-only deploy).

V19.70.5 plan:
- Client image upload UI in NewCampaignWizard
- Firebase Storage path: `campaigns/{userId}/{campaignId}/{filename}`
- Cron resolves URLs → base64 (or passes URL directly if bridge supports remote fetch)
- Bridge update + VPS redeploy
