## 🔧 التغييرات بالتفصيل

### 1. `src/utils/splitCollections.js` (موسّع)

**أُضيف:**
- `SPLIT_FIELDS_V1674` و `SPLIT_FIELDS_V1949` — تجميع الحقول حسب إصدار الـmigration
- `SPLIT_FLAG_V1674` و `SPLIT_FLAG_V1949` — flags للتحكم الانتقائي
- 4 collections جديدة في `SPLIT_COLLECTIONS`:
  - `custPayments → custPaymentsDays`
  - `supplierPayments → supplierPaymentsDays`
  - `wsPayments → wsPaymentsDays`
  - `checks → checksDays`

**اتعدّل:**
- `stripSplitArrays()` بقت **selective** — تحذف بس الحقول اللي migration بتاعتها انتهت (gated بـflag). ده بيمنع silent data loss في فترة rolling deploy.
- `readAllSplitCollections()` و `getAllSplitStats()` بقوا dynamic — يطلبوا كل الـSPLIT_FIELDS تلقائياً.
- `syncAllSplitChanges()` بقت تشتغل على كل الـSPLIT_FIELDS بدون hardcode.

### 2. `src/App.jsx` (موسّع)

كل المراجع الـ hardcoded للـ3 arrays القديمة (treasury/auditLog/hrLog) اتحوّلت لـ loops على `SPLIT_FIELDS`:

| المكان | قبل | بعد |
|---|---|---|
| `pendingSplitWritesRef` init | object literal بـ3 maps | `Object.fromEntries(SPLIT_FIELDS.map(f=>[f,new Map()]))` |
| `registerPendingSplitWrites` loop | `for(const f of ["treasury","auditLog","hrLog"])` | `for(const f of SPLIT_FIELDS)` |
| Stale cleanup interval | hardcoded 3 fields | `for(const f of SPLIT_FIELDS)` |
| Listener block `dayDocs` init | object literal | `Object.fromEntries(SPLIT_FIELDS.map(f=>[f,new Map()]))` |
| `firstFires` init | object literal | `Object.fromEntries(SPLIT_FIELDS.map(f=>[f,false]))` |
| `setSplitData(...)` shape | hardcoded 3 fields | `for(const f of SPLIT_FIELDS) nextSplit[f]=...` |
| `splitLoaded` trigger | `firstFires.treasury && ...auditLog && ...hrLog` | `SPLIT_FIELDS.every(f=>firstFires[f])` |
| `subscribeCol` calls | 3 explicit calls | `for(const f of SPLIT_FIELDS) subscribeCol(f, SPLIT_COLLECTIONS[f])` |
| `splitBefore` (upConfigTx) | object literal بـ3 fields | `Object.fromEntries(SPLIT_FIELDS.map(...))` |
| `explicitSplitBefore` (upConfig) | object literal بـ3 fields | `Object.fromEntries(SPLIT_FIELDS.map(...))` |
| Hydration in upConfig | hardcoded 3 fields | gated loops على V1674 + V1949 fields حسب الـflag |
| `newSplit` construction | hardcoded 3 fields | يبني فقط للحقول اللي flag بتاعتها on |
| `config` useMemo overrides | 3 explicit assignments | gated بـSPLIT_FLAG_V1674 + SPLIT_FLAG_V1949 |

**Migration جديد:**
بلوك useEffect كامل لـmigration V19.49 بنفس قالب V16.74:
1. ينتظر V16.74 يخلص (`_splitDaysV1674Done=true`)
2. يفحص الـflag `_splitDaysV1949Done` — لو موجود يخرج
3. يقفل UI بـ`setMigrationStatus`
4. ياخد backup كامل لـconfig في `backups/pre-migration-split-days-v1949-{ts}`
5. يكتب الـ4 arrays لـday docs المناسبة عبر `syncAllSplitChanges`
6. transaction atomic: يكتب الـflag الجديد + يحذف الحقول الـ4 من config
7. يكتب سجل في `migrationLog`
8. يعلن النجاح عبر `noticeSuccess`

## ⚙️ سلوك الـRuntime بعد التحديث

### السيناريو 1: تطبيق على بيانات V19.48 موجودة

1. المستخدم يفتح التطبيق على V19.49
2. `_splitDaysV1674Done=true` (من قبل) ، `_splitDaysV1949Done=undefined`
3. listeners تشتغل على كل الـ7 collections (3 قديمة + 4 جديدة) — الـ4 الجديدة تكون فاضية
4. config useMemo: V16.74 fields من splitData، V19.49 fields لسه من configDoc
5. Migration V19.49 يلاقي 4 arrays في configDoc → ينقلها لـday docs
6. Migration يحط الـflag الجديد + يحذف الحقول من configDoc (transaction atomic)
7. configDoc snapshot يصل من Firestore بدون الـ4 arrays
8. config useMemo بقى يعرضها من splitData (الـoverride تشتغل دلوقتي)
9. كل الصفحات (CustDeliverPg, PurchasePg, ExtProdPg) شغّالة عادي — **مفيش تعديل عليها**

### السيناريو 2: تطبيق على نسخة جديدة (no migration done yet)

1. _splitDaysV1674Done=false → V16.74 migration يشتغل أولاً
2. لما V16.74 يخلص → V19.49 migration يبدأ
3. النتيجة النهائية نفس السيناريو 1

### السيناريو 3: المستخدم يفتح التطبيق على V19.48 (rolling deploy)

1. configDoc من Firestore عنده `_splitDaysV1949Done=true` لكن V19.48 ميعرفش الـflag ده
2. **بدون selective stripping**: V19.48 كان هيكتب config لـFirestore بدون الـ4 arrays الجديدة (لأن الـlistener حضر الـ4 fields من الـconfig في الـcomponent state)
3. **مع selective stripping (V19.49)**: حقل لو فلاج بتاعته off (V19.48 ميعرفش يحط الـflag الجديد) فالحذف ميتمش — لكن مع ذلك في حالة rolling فيه risk على V19.48 clients
4. **التوصية:** زاكر deploy V19.49 لكل المستخدمين دفعة واحدة، مفيش mixed sessions
