### 🎯 Funnel Tab (Customer Lifecycle Pipeline)

**Reads from:**
- `data.customers` (real CLARK customer data) — for stage + tier distribution
- `data.aiAgentSuggestions` (empty until Phase D) — for pending transitions

**7 sections:**

1. **Header context** — total customer count, warning if any have no `stage` set yet
2. **Stage distribution** — proportional bars per stage (Stranger/Awareness/Interest/Decision/Customer/Repeat/Dormant) with count + %
3. **Movement آخر 7 أيام** — counts of stage transitions per from→to pair (computed from `customer.stage_history` where `stage_changed_at` ≥ 7d ago)
4. **Pending stage transitions** — list from `aiAgentSuggestions` filtered by `type === "stage_transition"` AND `status === "pending"`
5. **Auto-approve settings** — 6 transition toggles (with "high risk" badge for Decision-related)
6. **Tier distribution** — 4 cards (Bronze 🥉 / Silver 🥈 / Gold 🥇 / Platinum 💎) with counts + thresholds
7. **Approaching tier-up** — top 10 customers within 90% of next tier threshold

### 👥 Profiles Tab (Customer Profile Manager)

**Read from:** `data.customers`. **Writes to:** `data.customers` via `upConfig` (admin permissions).

**Filter bar:** search (name/phone/address) + stage dropdown + tier dropdown + 3 stat pills.

**Customer cards:** name + tier badge + stage badge + phone + address. Click → opens **Full Profile Modal**.

**Full Profile Modal sections:**

1. **Header** — name + tier badge + stage badge
2. **Phones** — primary + normalized + additional_phones list (with verified ✅ marker)
3. **AI Profile** — 8 fields (preferred categories/seasons/models, avg order value, total annual purchases, payment pattern, communication style, best response time). Empty fields show "—" + italic hint "هيـpopulate من Phase D"
4. **Admin Notes** — yellow tags, add/remove inline
5. **Observations** — AI-suggested OR admin-manual. Pending observations have ✅/❌ buttons; approved have only 🗑
6. **Flags** — 4 toggles (VIP, careful_handling, do_not_call, special_pricing)
7. **Stage History** — timeline of all stage changes

## ⚠️ Read-only carve-out (intentional)

The spec's READ-ONLY constraint applies to **the Agent backend** — it cannot write to `customers/`. The CLARK admin (via this UI) **can** write to customers (notes, observations, flags) — that's the existing CLARK security model and how admin notes have always worked.

The boundary is enforced at runtime in the backend's Firestore wrapper (Phase D). This UI is editing data on behalf of the admin user, not the Agent.
