## 🎯 الهدف

آخر array operational في `factory/config` (notifications) اتقسمت + اتعملها refactor مهم لحل race condition محتمل.

## 🔍 المشكلة التي V19.53 حلتها

```
قبل V19.53:
notifications: [
  {
    id: 123,
    msg: "...",
    readBy: [user1@, user2@],         // mutable!
    dismissedBy: [user3@],             // mutable!
    doneBy: [user4@],                  // mutable!
  }
]

السيناريو الخطر:
- 5 users يقروا نفس الإشعار في نفس اللحظة
- الـsplit engine: read day-doc → modify → write back
- 5 reads متوازية → 5 writes متوازية → race condition
- آخر write يفوز → readBy entries من users تانيين تتفقد
```

## ✅ الحل في V19.53

```
بعد V19.53:
notifications: [
  { id, msg, ... }    ← immutable post-create
]
+
userNotifStates/{userEmail}: {
  reads: { notifId: ts, ... },
  dismisses: { notifId: ts, ... },
  doneTasks: { notifId: ts, ... },
  updatedAt: ISO
}
```

كل user بيكتب فقط على doc بتاعه (`userNotifStates/{theirEmail}`).
- **صفر contention** بين users على نفس الـdoc
- الـnotification entry نفسه immutable → safe للـdaily split

## ⚙️ سلوك الـMigration

### السيناريو الطبيعي (production)

1. المستخدم يفتح التطبيق على V19.53
2. flags الموجودة: `_splitDaysV1674/V1949/V1950/V1952Done = true`. flag الجديد: `undefined`
3. listener جديد لـ`userNotifStates/{userEmail}` يبدأ (هيلاقي doc فاضي مبدئياً)
4. listener جديد للـnotificationsDays/* يبدأ (فاضي مبدئياً)
5. config useMemo: notifications لسه من configDoc
6. **Migration block يلاقي notifications array → بيـextract حالة كل user**
7. لكل user كان في `readBy`/`dismissedBy`/`doneBy` → setDoc(userNotifStates/{email}, {...}, merge:true)
8. الـnotifications تتنقل لـday docs بدون readBy/dismissedBy/doneBy
9. config.notifications اتمسحت + flag = true
10. مستخدمين مفتوحين على التطبيق هيشوفوا modal الـmigration → بيقفل تلقائي بعد 1.5s
11. كل user يفتح بعد كده هيـlistener على userNotifStates/{email} → الحالة المُهجّرة موجودة

### Edge cases

| الحالة | السلوك |
|---|---|
| Migration فشل | flag مش بيتكتب → retry تلقائي عند reload |
| User بيقرا إشعار قديم بعد migration | userNotifStates يتحدّث، الإشعار يختفي |
| User على V19.52 + user آخر على V19.53 | Filter logic فيها fallback للـreadBy/dismissedBy → يشتغل |
| API endpoint بيكتب notification بعد migration | Check flag → يكتب على notificationsDays |
| API endpoint قديم لسه نسخة قبل deploy | Push على config.notifications — لكن config.notifications مش موجود → setDoc بـmerge:true ينشئ array جديد. الـclient على V19.53 ميشوفوش لأن config useMemo override من splitData. **Fix: اتأكد كل الـAPI endpoints متحدّثين قبل الـmigration** |
