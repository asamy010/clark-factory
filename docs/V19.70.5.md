## 1. Check Payment Received trigger

### New event type: `checkPaymentReceived`

نفس الـUX pattern كـpaymentReceived بس مع check details. Recipients: customer + owner.

**Variables (per role):**
- customer: `{customerName} {amount} {bank} {checkNo} {dueDate} {batchInfo} {balance} {date}`
- owner: نفس الـcustomer + `{office}`

**`{batchInfo}` semantic:**
- Single check: `""` (empty)
- Batch (حافظة): `"(شيك X من Y)"` — e.g. `"(شيك 1 من 3)"`

### Default templates

**Customer:**
```
🏦 *تم استلام شيك* (شيك 1 من 3)

البنك: بنك مصر
رقم الشيك: 12345678
القيمة: 5,000 ج.م
تاريخ الاستحقاق: 2026-05-05
الرصيد المتبقي: 7,500 ج.م

شكراً لك 🌟
```

**Owner:**
```
🏦 *شيك من عميل* (شيك 1 من 3)

أحمد محمد — مؤسسة الأمل للملابس
البنك: بنك مصر
الشيك: 12345678
القيمة: 5,000 ج.م
الاستحقاق: 2026-05-05
الرصيد المتبقي: 7,500 ج.م
```

### Detection

**Client-side instant** (TreasuryPg check form save):
- Fires only when: NEW (not edit) + type=receivable + category=دفعة عميل + linkedCustId + customer has phone
- Pre-generates IDs for each check in batch
- Fire-and-forget per check (one fetch call per check) with `idempotencyKey: "checkPay:${id}"`
- Customer balance computed client-side (orders − returns − cash payments — checks are neutral)

**Cron-side fallback** (`scanRecentChecks` in automation-tick):
- Filter: `c.type === "receivable" && c.status === "معلق" && c.category === "دفعة عميل"`
- Recency: last 24h based on `c.date` (user-facing date)
- Backfill prevention: skip if `c.createdAt < eventCfg.enabledAt`
- Same `idempotencyKey: "checkPay:${id}"` — client and cron never double-fire

### Per-check messages for batches

If user adds a حافظة of 3 checks, the customer receives:
1. Message 1: "🏦 *تم استلام شيك* (شيك 1 من 3) ..."
2. Message 2: "🏦 *تم استلام شيك* (شيك 2 من 3) ..."
3. Message 3: "🏦 *تم استلام شيك* (شيك 3 من 3) ..."

Each message has its own check details (own checkNo, own dueDate if step is set, etc.).

## 2. Image campaigns

### Bridge already supports media

Discovered during implementation: `clark-wa-bridge/server.js` (V19.31) already has full `MessageMedia` support. Per-message format:

```js
{ phone, message, media: [{base64, mime, name}] }
```

The bridge handles the base64→`MessageMedia.fromUrl/fromBase64` conversion + sends via whatsapp-web.js. **No VPS redeploy needed.**

### Client-side image picker

In ChooseSendMode → "📅 جدولة لوقت لاحق":

- File input: `<input type="file" accept="image/*" multiple>`
- Compression: canvas resize to 1024px (long side) + JPEG quality 0.7
- Validation: max 4 images per campaign, max 200KB per image (after compression)
- Preview: 80×80 thumbnails with × remove button + size badge
- Storage: inline base64 in `data.scheduledCampaigns[].images[]`

### Storage size budget

- Per image after compression: ~80–200KB
- Per campaign with 4 images: ~800KB max
- Total scheduledCampaigns cap: 100 entries (existing)
- Inline storage in `factory/config` — fits in 1MB Firestore doc limit if user keeps active scheduledCampaigns small

For users with many active scheduled campaigns + heavy images, V19.71 may add Firebase Storage offload.

### Cron firing with images

`scanScheduledCampaigns` builds the bridge payload:

```js
const mediaPayload = images
  .filter(img => img && img.base64 && img.mime)
  .map(img => ({ base64: img.base64, mime: img.mime, name: img.name }));

const messages = items.map(it => {
  const msg = { phone: it.phone, message: personalize(template, it) };
  if (mediaPayload.length > 0) msg.media = mediaPayload;
  return msg;
});
```

Same images attached to every recipient.

## Known limitations / V19.71 candidates

1. **Image storage** — inline base64 limits campaigns to ~800KB. For users wanting many images or larger, V19.71 may use Firebase Storage with cron-side download.
2. **Per-customer opt-out** — still no way for a customer to say "don't message me". Future feature.
3. **Image-on-cash payments** — only campaigns can attach images. paymentReceived/saleCompleted/checkPaymentReceived are text-only events. Could be extended.
4. **Portal links in cron-fired campaigns** — `{لينك}` placeholder skipped (needs admin SDK signing). Manual-mode campaigns still get portal links via the existing `portalUrlBatch` flow.
