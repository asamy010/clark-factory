## Issue #1: Payment WhatsApp message still duplicated (V19.76.3 not enough)

### السياق

User report (after V19.76.3 atomic-claim shipped):
> "الدفعة برضه بتتكرر، بعد تسجيلها اتبعتت مرتين"

V19.76.3 لـ atomic claim بـ idempotencyKey كان بيقفل الـ race بين الـ client
instant fire والـ cron tick. الـ claim transaction بـ يكتب `inFlight: true` قبل
الـ bridgeSend، وأي caller تاني بنفس الـ key بـ يـ dedup.

لكن الـ user لسه بـ يـ شوف نسختين. سيناريوهات ممكنة:
1. **Sync race** — الـ payment اتسجل مرتين في `custPayments` بـ IDs مختلفة (rare
   Firestore retry / multi-tab race) → idempotencyKeys مختلفة → claim ما يـ catch
2. **Edit re-fire** — الـ user عدل الدفعة (edit يـ replace الـ id بـ gid() جديد)
   → الـ cron يـ pick up الـ new id → يـ fire رسالة شكر تانية
3. **Force bypass** — caller بـ `force: true` (manual replay) بيـ skip الـ claim
4. **Bridge-side retry** — الـ bridge عنده retry داخلي بيرسل رسالتين

### الإصلاح: Content-based dedupe (last-resort safety net)

أضيف safety net جديد في `claimEvent` — حتى لو الـ idempotencyKey مختلف، لو
المحتوى متطابق + نفس الـ recipient phone خلال آخر **30 ثانية**، الـ claim يرفض.

```js
const CONTENT_DEDUPE_MS = 30_000;

const recipPhone = customerPhone || supplierPhone || "";
const contentSig = recipPhone
  ? eventType + "|" + recipPhone + "|" + summarizePayload(eventType, payload)
  : null;

// Inside the transaction:
if (contentSig) {
  const collision = history.find(h => {
    if (h.idempotencyKey === idempotencyKey) return false; // skip self
    const sig = h.contentSig || (...); // fallback for legacy entries
    if (sig !== contentSig) return false;
    const ageMs = Date.now() - Date.parse(h.at || 0);
    return ageMs < CONTENT_DEDUPE_MS;
  });
  if (collision) return { claimed: false, reason: "content-duplicate" };
}
```

`summarizePayload` بـ يرجع نفس الـ string لو نفس الـ key fields (e.g.
`paymentReceived`: customerName + amount). فلو الـ user حفظ دفعة، اتعملت entry،
وبعد ثانية اتعملت entry تانية بنفس الـ amount لنفس الـ customer → الـ claim
بـ يرفض الـ second بسبب الـ contentSig match.

**Owner-only events (مفيش phone للعميل/المورد)** بـ يـ skip الـ check (recipPhone
فاضي → contentSig null) — الـ owner ممكن يحب يستلم نسختين لأي سبب.

الـ recordResult بـ يـ persist الـ recipPhone + contentSig في الـ final entry
بعد الـ success، فالـ window الـ 30 ثانية يفضل ساري.

### Diagnostic logging

أي dedupe بقى يطبع log line في Vercel:
```
[event-trigger] DEDUPED { eventType, idempotencyKey, reason, source }
[event-trigger] FORCE bypass — claim skipped { ... }
[event-trigger] FIRING bridge { eventType, idempotencyKey, source, recipientCount }
```

لو الـ user لسه بـ يشوف duplicate بعد V19.76.8، الأدمن يقدر يـ grep الـ Vercel
logs ويعرف:
- هل الـ second fire وصل أصلاً (لو لأ → الـ bridge بـ يكرر داخلياً)
- هل dedupe ضرب (لو نعم بس فيه duplicate → bridge issue)
- هل caller بـ `force=true` (لو نعم → manual replay)

## Issue #2: Replace native browser confirm() with themed popup

### السياق

User screenshot showed `clark-factory.vercel.app says: هتمسح 3 شيك. متأكد؟` —
الـ native browser dialog. The user said:
> "عاوز ده بوب اب مش كده"

native confirm() مش consistent مع الـ UI الباقي:
- بـ يعرض الـ URL فوق العنوان
- مش themed (light/dark mode غير مدعوم)
- مش RTL
- بـ يقفل الـ JS thread

### الإصلاح

استبدلت كل الـ 6 استخدامات بـ `ask()` المخصصة (في `src/utils/popups.js`):

| File | Use case |
|---|---|
| `TreasuryPg.jsx` | حذف الشيكات المحددة (bulk) |
| `AutomationPg.jsx` | (1) إلغاء lastSentAt للـ scheduler test (2) إرسال كل الـ pending (3) حذف pending entry |
| `CampaignsPg.jsx` | (1) إلغاء حملة مجدولة (2) حذف حملة نهائياً |
| `TasksPg.jsx` | حذف مهام البوت القديم |

كل واحد بـ `danger:true` للحذف الخطير، و `confirmText` مخصص (e.g. "🗑 حذف").
