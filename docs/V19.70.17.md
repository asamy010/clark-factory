# V19.70.17 — PDF attachment toggle in bulk WhatsApp send (workaround)

**Date:** 2026-05-06
**Type:** feature, ux (workaround for the open Arabic PDF issue)
**Status:** shipped

## Why this version exists

Three attempts to fix the Arabic glyph shaping in the bulk-WA-send PDF have not held:
- **V19.70.14** — `<link>` injection + `document.fonts.load(string)` (race condition)
- **V19.70.15** — explicit `FontFace` API + WOFF2 binaries (html2canvas iframe doesn't see registered FontFaces)
- **V19.70.16** — switch `<th>` to system fonts (Tahoma, Arial) — also didn't render correctly

The user's verdict: "نفس المشكلة ماتحلتش - كده مش مشكلة فونت." Even system fonts that are pre-loaded by the OS render incorrectly when captured by html2canvas. This points to a fundamental limitation of html2canvas with RTL/Arabic, not specific to font loading.

The user's pragmatic decision: stop trying to fix the PDF immediately, and add a UI toggle so the user can send text-only WhatsApp messages without a broken PDF reaching customers.

## What V19.70.17 ships

### UI: new checkbox in the bulk-print popup

Below "📦 إذن تسليم بدون أسعار", a new toggle:
- **`📎 إرفاق نسخة PDF مع رسالة الواتس`**
- Default: **OFF** (text-only is the safer mode while the PDF is broken)
- WhatsApp green (`#25D366`) styling when on, neutral when off
- Subtitle adapts to state:
  - ON: "PDF + رسالة تفاصيل لكل عميل"
  - OFF: "رسالة تفاصيل نصية فقط (الموصى به مؤقتاً — حتى يتم حل مشكلة الخط العربي في الـPDF)"

### Behavior changes

When `includePdf = true` (existing behavior):
- Load `html2canvas` + `jsPDF` from CDN.
- Pre-fetch `/api/delivery-sign` for QR signatures.
- Build HTML → render PDF → attach as `media`.
- Message includes the QR confirmation note ("📱 برجاء مسح كود QR...").

When `includePdf = false` (new default):
- Skip the PDF library load (~200KB CDN download saved).
- Skip the signature fetch (the QR isn't needed — no PDF).
- Skip the html2canvas capture (~3-5 seconds saved per customer).
- Message includes the customer's **phone + address** so it's self-contained.
- The QR confirmation note is dropped (it's irrelevant without the PDF).
- The bridge `/send` payload omits the `media` field entirely.

### Confirmation prompt adapts

Before:
> "هيتم إرسال إذن استلام (PDF) + رسالة تفاصيل لكل عميل..."

After (conditional):
- PDF on: same as before.
- PDF off: "هيتم إرسال رسالة تفاصيل (نصية فقط — بدون PDF) لكل عميل..."

The user knows exactly what reaches the customer before clicking through.

## Files touched

- `src/pages/CustDeliverPg.jsx` — `doSendWhatsApp` function + popup UI

## Performance impact

In text-only mode, the bulk send is much faster:
- No CDN download for the PDF libs (~200KB).
- No html2canvas capture (~3-5 seconds per customer).
- No signature round-trip (~1 network call).

For a batch of 50 customers, the total send time drops from ~10-15 minutes to ~5-8 minutes. The delays between sends are still preserved (bridge anti-ban).

## Path forward for the underlying PDF bug

If the user wants the PDF feature back eventually, the options are (in order of effort):

1. **jsPDF text APIs with embedded TTF** (Approach A from the V19.70.14 handoff) — refactor `htmlToPdf.js` to draw text via jsPDF's text engine instead of html2canvas image capture. Reliable for Arabic but a substantial rewrite of the PDF generator.

2. **Server-side Puppeteer** (Approach B) — add a Vercel function `/api/render-pdf` that uses `puppeteer-core` + `@sparticuz/chromium`. Server-side font loading is bulletproof. Adds ~3-5 seconds per PDF and dependencies.

3. **html2pdf.js with different config** — try `html2pdf.bundle.min.js` (a wrapper around html2canvas + jsPDF with opinionated defaults). Same underlying engine, so same risk.

Option 1 is the proper long-term fix. The V19.70.17 toggle is a workaround that lets the user keep using the bulk send productively in the meantime.
