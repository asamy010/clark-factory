# V19.70.11 — checkEndorsed + checkRePresented + bounced→re-bounce supported

**Date:** 2026-05-05
**Type:** feature, ux
**Status:** shipped

## Events added

### `checkEndorsed` — شيك مُظهَّر لمورد
When a check is endorsed from a customer to a supplier (`status=مُظهَّر`), the supplier receives an instant message with full check details and the original customer's name (the drawer) for traceability.

Payload: `customerName` (drawer), `supplierName` (recipient), `bank`, `checkNo`, `amount`, `dueDate`, `customerOffice`, `supplierOffice`, `balance` (the supplier debt, reduced by the check value because we owe them less now).

Hook location: `endorseCheck()` in TreasuryPg — snapshot the check + customer BEFORE `upConfig`, fire AFTER.

### `checkRePresented` — إعادة تقديم شيك مرتد
The existing UI button `↻ إعادة` (transitions `status=مرتد→معلق`) now fires an event for the customer + owner.

Customer message: `إعادة تقديم شيك للبنك` with the original date, the re-presentation date, and the outstanding balance (decreases by the check value again because the check is active in the receivable cycle once more).

Transition detection in `updateStatus`: `prevStatus === 'مرتد' && newStatus === 'معلق'`.

## Re-bounce support — date-suffixed idempotency keys

If a check bounces → gets re-presented → bounces again, V19.70.10's `checkBounced:${id}` would have been deduplicated by `eventHistory`. The 2nd bounce wouldn't fire.

**Fix:** idempotency keys are now date-suffixed:
- `checkBounced:${id}:${dt}`
- `checkRePresented:${id}:${dt}`

Each bounce/re-present fires successfully even if it's the 2nd or 3rd round on the same check.
