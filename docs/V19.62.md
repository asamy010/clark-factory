## 🎯 The Real Root Cause

**الـbug ده عمره من V19.57.** كل المحاولات السابقة كانت بتـpatch downstream symptoms بدون ما تمسك الـupstream wipe.

### السطر المسؤول (App.jsx:3229 pre-V19.62)

```js
const explicitPartBefore={
  hrWeeks:[...(partitionedDataRef.current.hrWeeks||[])],
};
```

السطر ده اتكتب في **V16.75** لما hrWeeks كان الـpartitioned الوحيد. لما V19.57 ضافت 8 master collections (customers, suppliers, workshops, employees, empDebts, generalProducts, fabrics, accessories)، السطر ده **ما اتحدّثش**. والباقي اتحدّث:

| Pattern | السطر | اتعمل dynamic؟ |
|---|---|---|
| `explicitSplitBefore` | 3226 | ✅ Object.fromEntries(SPLIT_FIELDS.map(...)) |
| **`explicitPartBefore`** | **3229** | ❌ **hardcoded {hrWeeks: ...}** |
| `upConfigTx` partBefore fallback | 3030 | ✅ Object.fromEntries(PARTITIONED_FIELDS.map(...)) |
| `explicitSalesSplitBefore` | 3493 | ✅ dynamic |
| `explicitTasksSplitBefore` | 3640 | ✅ dynamic |

الـ4 الباقيين dynamic. الـpartitioned فقط نُسي.

## 🔬 الـCausal Chain (Sale → Wipe)

1. User يأكد بيع → `upConfig(fn)` يتنادى
2. **Line 3229**: `explicitPartBefore = {hrWeeks: [...]}` — مفيش customers/suppliers/...
3. **Line 3299** (hydration loop):
   ```js
   for(const f of PARTITIONED_FIELDS_V1957){
     next[f] = JSON.parse(JSON.stringify(explicitPartBefore[f]||[]))
              // explicitPartBefore.customers = undefined → []
   }
   ```
   → `next.customers = []`, `next.suppliers = []`, إلخ
4. `fn(next)` (logic البيع) ميـلمسش customers → يفضلوا `[]`
5. **Line 3329-3332**: `newPart.customers = []` (لكل master field)
6. **Line 3355-3356**:
   ```js
   setPartitionedData(newPart);            // state.customers = []!
   partitionedDataRef.current = newPart;   // ref.customers = []
   ```
7. **القائمة تختفي في الـUI** ← الـsymptom اللي اتبلّغ

### ليه Firestore data بقى سليم

في `syncAllPartitionedChanges`:
```js
if (oldArr.length === newArr.length && _deepEqual(oldArr, newArr)) continue;
```
`partBefore.customers = undefined → []`، `partAfter.customers = []`. الاتنين فاضيين → SKIP، مفيش Firestore writes. **الـserver-side البيانات محفوظة.**

عشان كده hard refresh يرجع البيانات (الـlistener يـfire من Firestore cache بعد re-mount). بس بمجرد بيع تاني → wipe تاني.

### ليه `isSafeWrite` ميمسكش الـbug ده

`isSafeWrite(prev, next)` بيقارن configDoc-form fields. post-V19.57 strip، prev.customers = undefined و next.customers = []. الـcheck `before > 0 && after === 0` بيـevaluate لـfalse (before=0). الـguard كان مصمم للـlegacy schema، مش الـpartitioned schema.

## ✅ الإصلاحات في V19.62

### Fix #1: Dynamic explicitPartBefore (الـcritical)

```js
// قبل (V19.57 → V19.61):
const explicitPartBefore={
  hrWeeks:[...(partitionedDataRef.current.hrWeeks||[])],
};

// بعد (V19.62):
const explicitPartBefore=Object.fromEntries(
  PARTITIONED_FIELDS.map(f=>[f,[...(partitionedDataRef.current[f]||[])]])
);
```

نفس الـpattern في الـ4 الباقيين. الـsnapshot دلوقتي يحتوي على كل الـ9 partitioned fields (1 V1675 + 8 V1957).

### Fix #2: Defensive Safety Guard

ضافت check في upConfig قبل الـwrites:

```js
if(newPart){
  const wipes=[];
  for(const f of PARTITIONED_FIELDS_V1957){
    const before=(partitionedDataRef.current[f]||[]).length;
    const after=(newPart[f]||[]).length;
    if(before>=5&&after===0)wipes.push(`${f}: ${before} → 0`);
  }
  if(wipes.length>0){
    console.error("[V19.62 BLOCKED] Refusing partitioned master-data wipe:",wipes,...);
    showToast("⛔ تم منع تعديل خطير: ممكن يمسح بيانات master ...");
    return;
  }
}
```

**Threshold ≥5**: آمن من false positives (single edits ميـtripش)، عالي من any silent regression. mass-delete UI مش موجود في الـapp اليوم — أي trip = bug.

ده يمنع نفس النوع من الـbugs (hardcoded snapshots عند إضافة fields جديدة) من الـsilent escalation مستقبلاً.

## 🧠 الـLessons Learned

1. **3 race-condition fixes متتاليين بدون نجاح = الـbug مش race**. الـpattern ده كان signal واضح للـrethink.
2. **الـdiagnostic الحاسم**: لما symptom = "data disappears from state"، اعمل grep لكل `setX(`. لو في 2 places، فحص الـ2. هنا الـlistener rebuild اتحمى في V19.60/61 لكن `upConfig` كان مكشوف.
3. **Hardcoded snapshots عند الـmutation = code smell**. لو ضافت field جديد للـschema، الـsnapshot المقابل لازم يتحدّث. الـsafety guard في V19.62 يـcatch ده مستقبلاً.
4. **Symptoms ميقولش الـsource**. تشخيصات V19.59/60/61 كلها كانت صحيحة (real races موجودة)، لكن مكنوش هم الـsource. السلسلة من الـpatches في حالة symptom-only كانت إشارة كافية للرجوع للـoriginal architecture review.
