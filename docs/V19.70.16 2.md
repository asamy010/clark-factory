# V19.70.16 — Final fix: Arabic table headers via system fonts

**Date:** 2026-05-06
**Type:** fix (final)
**Status:** shipped — pending user verification on real device

## Why V19.70.15 didn't fully fix it

V19.70.15 switched from `document.fonts.load(string)` to the explicit `FontFace` API + WOFF2 binaries from fontsource CDN. The face was successfully loaded and registered to `document.fonts`. However, the user confirmed: **headers were still scrambled**.

Root cause (third-time-lucky understanding):

> **html2canvas clones the DOM into an internal iframe** to capture pixels. The `document.fonts` registry of the parent page is NOT reliably copied to that iframe. Even though we registered Cairo Bold via `FontFace` and `document.fonts.add()`, html2canvas's internal clone could not see it. The iframe fell back to Arial, which doesn't shape Arabic ligatures correctly.

This is a known limitation of html2canvas with custom web fonts. There are workarounds (e.g. injecting `@font-face` rules with base64 data URLs into the cloned styles) but they're fragile and bundle-heavy.

## V19.70.16's fix (user's idea)

The user suggested: just use Arial for the table headers. That bypasses the entire problem because system fonts are pre-loaded by the OS — html2canvas's iframe inherits them automatically without any FontFace registration.

We went one step further with a 3-layer fallback chain:

```css
th {
  font-family: 'Tahoma', 'Arial', sans-serif;
  font-weight: 700;  /* down from 800 — Tahoma's bold is 700 */
  /* ...rest unchanged */
}
```

- **Tahoma** — preinstalled on every Windows version since Win 95, has full Arabic glyph shaping support. Best default.
- **Arial** — preinstalled on macOS, has Arabic shaping. The user's explicit suggestion.
- **sans-serif** — generic fallback. Browsers map this to a system Arabic-aware font on every platform (Noto Sans Arabic on Linux, etc.).

The `<td>` body cells keep using Cairo (font-weight: 400). That weight loaded correctly all along — the bug was specifically Cairo Bold (weight 800) inside the html2canvas clone. With the body still on Cairo, the brand identity is preserved where it matters most (customer name, money values, item descriptions).

## Visual impact

- Headers render slightly thinner than before (Tahoma 700 < Cairo 800).
- Headers slightly more compact horizontally (Tahoma is denser than Cairo).

The receipt is still recognizably "CLARK styled" because the brand colors, layout, and body font haven't changed.

## Lesson for future work

If a PDF requires complex script support (Arabic, Hebrew, Devanagari) and is generated via html2canvas:
- **Use system fonts** (Tahoma, Arial, Times New Roman) for elements that contain that script.
- **Avoid web fonts** for those elements — html2canvas + FontFace API is unreliable.
- Web fonts are still safe for elements that contain only Latin script (numbers, English text).

Alternative approach for future: use jsPDF text APIs directly with embedded TTF (Approach A from the V19.70.14 handoff). That eliminates html2canvas entirely.
