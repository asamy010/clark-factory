## ✅ ما تم إنجازه

### 1. دوال Stock Management المركزية (100 سطر)

#### `calcStockNeeded(order)`
ترجع **ما يحتاجه** الأوردر من خامات وإكسسوارات:
```javascript
{
  fabrics: {'fab-1': 80, 'fab-2': 50},     // cons × cutQty
  accessories: {'acc-1': 500, 'acc-2': 100}, // qtyPerPiece × cutQty
  cutQty: 100
}
```

#### `checkStockAvailability(order, data)`
تفحص المخزن وترجع:
```javascript
{
  ok: false,
  shortages: [
    { itemType: "fabric", itemName: "قطن أزرق", 
      needed: 80, available: 50, shortage: 30 }
  ]
}
```

**ذكية بـ Delta:**
- لو الأوردر اتقص قبل كده → تشيك الزيادة بس
- لو الـ cons نقص → ترجع الفرق للمخزن (delta سالب)

#### `deductStockForOrder(d, order, userName)`
تخصم (أو ترجع) من المخزن بـ 3 سيناريوهات:

1. **زيادة** (delta > 0): `stock -= delta`, type="out"
2. **نقصان** (delta < 0): `stock += |delta|`, type="in" (إرجاع)
3. **حذف خامة** من الأوردر: `stock += old_qty` (إرجاع كامل)

تحفظ snapshot `_stockDeducted` على الأوردر للحسابات القادمة.

### 2. Hard Block في 3 أماكن

#### ✅ `addOrder` (أوردر جديد)
```
مستخدم يحفظ أوردر جديد
→ calcStockNeeded → checkStockAvailability
→ إذا ناقص: popup خطأ بتفاصيل كل صنف ناقص + ⛔ منع الحفظ
→ إذا كافي: حفظ + خصم تلقائي + stockMovements
```

#### ✅ `replaceOrder` (تعديل أوردر)
```
مستخدم يعدل cons أو cutQty أو يضيف/يحذف إكسسوار
→ حساب الـ delta
→ لو الزيادة تتخطى المتاح: ⛔ منع + عرض الناقص
→ لو الـ delta سالب: يرجع للمخزن تلقائياً
```

#### ✅ `delOrder` (حذف أوردر)
```
مستخدم يحذف أوردر
→ لو كان _stockDeducted: يرجع كل الخامات للمخزن
→ stockMovements بـ notes "إرجاع — حذف الأوردر"
```

### 3. Popup "المخزن غير كافي" (Arabic + Detailed)

```
⛔ المخزن غير كافي:

  المطلوب: 80 متر | المتاح: 50 متر | ناقص: 30 متر

  المطلوب: 500 قطعة | المتاح: 300 قطعة | ناقص: 200 قطعة

```

يظهر بـ `tell()` type="error" — متسق مع باقي الـ UX.

### 4. Delta Logic مع 4 سيناريوهات

| السيناريو | قبل | بعد | Delta | التصرف |
|---|---|---|---|---|
| زيادة cons | 0.8 | 0.9 | +10 متر | شيك توفره، خصم من المخزن |
| نقصان cutQty | 100 | 50 | -40 متر | إرجاع 40 للمخزن + stockMovement in |
| حذف خامة | fab-A | (مش موجودة) | إرجاع كامل | القيمة القديمة ترجع للمخزن |
| إضافة خامة | — | fab-B جديد | +كل الكمية | شيك + خصم |

### 5. Pre-Activation Safeguard

الأوردرات اللي تاريخها **قبل** `stockActivationDate`:
- ما يتمش شيك المخزن
- ما يتمش خصم
- تقدر تعدل فيها بدون مشاكل

هذا **ضروري** عشان الأوردرات القديمة قبل التفعيل.

## 🔒 الأمان والسلامة

### ✅ مفيش خصم غير مصرح
- `stockEnabled=false` → مفيش أي خصم (backward compat)
- `autoDeductOnCut=false` → مفيش خصم حتى لو enabled
- الأوردرات القديمة (قبل activation) → مستثناة

### ✅ Transaction-safe
- الـ `addDoc` + `upConfig` بيحصلوا على التوالي
- لو الـ stock check فشل: ما يتحفظش الأوردر أصلاً
- لو الـ upConfig فشل: الأوردر اتحفظ لكن المخزن لسه (نادر — نحتاج rollback manual)

### ✅ Delta-aware
- `_stockDeducted` snapshot بيمنع الـ double-deduction
- تعديل الأوردر = delta بس، مش خصم كامل جديد

### ✅ Human-readable Movement Notes
- "قص موديل SH-123"
- "قص موديل SH-123 (تعديل — إرجاع للمخزن)"
- "إرجاع خامة من موديل SH-123 (تعديل)"

## ⚠️ حالات Edge الحذرة

### الحالة 1: مستخدم بيعدل في OrdForm ومفيش مخزن
- `stockEnabled=false` → **مفيش شيك، مفيش خصم** (آمن)

### الحالة 2: تعديل أوردر قديم بعد التفعيل
- `order.date < activationDate` → **مستثنى من الشيك والخصم**
- عدّل بحرية بدون تأثير على المخزن

### الحالة 3: حذف أوردر ما اتخصمش منه
- `_stockDeducted = undefined` → الحذف يمشي طبيعي بدون إرجاع

### الحالة 4: حذف فابريك من الـ DB لسه موجود في الأوردر
- الـ movement بيتسجل بس الـ stock ما يتحدثش (idx<0)
- مفيش crash — يتسكت

### الحالة 5: Order بدون cutQty
- `cutQty<=0` → needed فارغ → مفيش check، مفيش deduct
